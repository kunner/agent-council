## 1. Level 1: 사용자 인증 (Firebase Auth)

### 1.1 개요

Agent Council의 1차 인증 계층은 Firebase Authentication을 사용한다. 사용자는 Google 또는 GitHub 소셜 계정으로 로그인하며, Firebase가 발급한 ID Token을 모든 서버 API 요청의 Bearer 토큰으로 사용한다.

```
┌─────────────────────────────────────────────────────────────┐
│                   인증 흐름 개요                              │
│                                                              │
│  Browser ──→ Firebase Auth (Google/GitHub OAuth)            │
│                     │                                        │
│                     ↓                                        │
│            Firebase ID Token 발급                            │
│                     │                                        │
│         ┌───────────┴───────────┐                           │
│         ↓                       ↓                           │
│  Firestore 직접 접근         Council Server API              │
│  (Security Rules 검증)       (Bearer 토큰 검증)              │
```

```typescript
// packages/web/src/firebase/auth.ts
import { getAuth, GoogleAuthProvider, signInWithPopup } from 'firebase/auth'

const auth = getAuth()
const googleProvider = new GoogleAuthProvider()

// 권장 스코프: 기본 프로필 + 이메일만 요청 (최소 권한 원칙)
googleProvider.addScope('profile')
googleProvider.addScope('email')

export async function signInWithGoogle(): Promise<UserCredential> {
  try {
    const result = await signInWithPopup(auth, googleProvider)
    // result.user: Firebase User 객체
    // result.user.uid: Firestore 문서 키로 사용
    await ensureUserDocument(result.user)
    return result
  } catch (error) {
    if (error.code === 'auth/popup-closed-by-user') {
      // 사용자가 팝업을 닫은 경우 — 조용히 처리
      return null
    }
    throw error
  }
}

async function ensureUserDocument(user: FirebaseUser): Promise<void> {
  const userRef = doc(db, 'users', user.uid)
  const snap = await getDoc(userRef)
  if (!snap.exists()) {
    await setDoc(userRef, {
      displayName: user.displayName,
      email: user.email,
      photoURL: user.photoURL,
      authMethod: 'api_key',   // 기본값, 설정에서 변경 가능
      createdAt: serverTimestamp(),
    })
  }
}
```

```typescript
// packages/web/src/firebase/auth.ts
import { GithubAuthProvider, signInWithPopup } from 'firebase/auth'

const githubProvider = new GithubAuthProvider()
// GitHub 소셜 로그인 스코프: 기본 프로필만 (repo 접근은 PAT로 별도 처리)
// GitHub PAT는 Level 3에서 별도 등록

export async function signInWithGitHub(): Promise<UserCredential> {
  try {
    const result = await signInWithPopup(auth, githubProvider)
    await ensureUserDocument(result.user)
    return result
  } catch (error) {
    // GitHub 계정이 이미 다른 제공자로 연결된 경우
    if (error.code === 'auth/account-exists-with-different-credential') {
      // 기존 제공자로 먼저 로그인 후 계정 연결 유도
      throw new Error('이 이메일은 다른 로그인 방법으로 이미 등록되어 있습니다.')
    }
    throw error
  }
}
```

#### 주의: GitHub OAuth vs GitHub PAT 구분

| 용도 | 방법 | 저장 위치 |
|---|---|---|
| 사용자 신원 확인 (로그인) | GitHub OAuth (Firebase Auth) | Firebase Auth |
| GitHub API 호출 (PR 생성 등) | Personal Access Token (PAT) | Firestore 암호화 저장 |

Firebase Auth의 GitHub OAuth는 **로그인 신원 확인 전용**이다. GitHub API 호출(브랜치 생성, PR 생성 등)에는 Level 3에서 별도 등록하는 PAT를 사용한다.

### 1.4 세션 관리

#### Firebase ID Token 구조

Firebase ID Token은 JWT(JSON Web Token) 형식이다.

```
Header.Payload.Signature

Payload 예시:
{
  "iss": "https://securetoken.google.com/{project-id}",
  "aud": "{project-id}",
  "auth_time": 1742720000,
  "user_id": "abc123uid",
  "sub": "abc123uid",
  "iat": 1742720000,
  "exp": 1742723600,      ← 발급 후 1시간 유효
  "email": "user@example.com",
  "email_verified": true,
  "firebase": {
    "sign_in_provider": "google.com"
  }
}
```

#### 토큰 갱신 전략

Firebase ID Token의 유효기간은 **1시간**이다. 클라이언트는 토큰 만료 전에 자동으로 갱신해야 한다.

```typescript
// packages/web/src/hooks/useAuthToken.ts
import { getAuth } from 'firebase/auth'

/**
 * 서버 API 호출 시 항상 이 함수로 토큰을 가져온다.
 * forceRefresh: false → 캐시된 토큰 반환 (만료 전이면 기존 토큰)
 * forceRefresh: true  → 항상 새 토큰 발급 (만료 직전 호출 등)
 */
export async function getIdToken(forceRefresh = false): Promise<string> {
  const user = getAuth().currentUser
  if (!user) throw new Error('로그인이 필요합니다.')
  return user.getIdToken(forceRefresh)
}

/**
 * onIdTokenChanged 리스너로 토큰 갱신을 자동 감지
 * Firebase SDK가 백그라운드에서 자동 갱신 처리
 */
export function useAuthToken() {
  const [token, setToken] = useState<string | null>(null)

  useEffect(() => {
    const unsubscribe = getAuth().onIdTokenChanged(async (user) => {
      if (user) {
        const newToken = await user.getIdToken()
        setToken(newToken)
      } else {
        setToken(null)
      }
    })
    return unsubscribe
  }, [])

  return token
}
```

#### 세션 타임아웃 정책

```
┌─────────────────────────────────────────────────────────────┐
│                  세션 상태 전환                               │
│                                                              │
│  로그인 완료                                                  │
│      ↓                                                       │
│  ID Token 유효 (1시간)                                        │
│      ↓  (만료 전 자동 갱신 — Firebase SDK 처리)               │
│  Refresh Token으로 새 ID Token 발급                           │
│      ↓                                                       │
│  로그아웃 또는 계정 비활성화 시 Refresh Token 무효화            │
│                                                              │
│  보안 이벤트 발생 시 (비밀번호 변경 등):                        │
```

Firebase의 Refresh Token은 명시적 로그아웃 전까지 유효하다. 장기 미사용 세션은 애플리케이션 레벨에서 처리한다.

```typescript
// packages/web/src/stores/authStore.ts — 비활성 세션 관리
const INACTIVITY_TIMEOUT_MS = 7 * 24 * 60 * 60 * 1000  // 7일

export function setupInactivityCheck() {
  const lastActivity = localStorage.getItem('lastActivity')
  if (lastActivity) {
    const elapsed = Date.now() - parseInt(lastActivity)
    if (elapsed > INACTIVITY_TIMEOUT_MS) {
      getAuth().signOut()
      localStorage.removeItem('lastActivity')
      return
    }
  }
  // 활동 시 타임스탬프 갱신
  document.addEventListener('click', () => {
    localStorage.setItem('lastActivity', Date.now().toString())
  })
}
```

### 1.5 클라이언트 ↔ 서버 인증 (Bearer Token 검증)

#### 요청 흐름

```
│                                                               │
│  Client                         Council Server               │
│                                                               │
│  const token = await getIdToken()                            │

│                                              │                │
│                                         유효한 토큰?          │

│                                         └─ NO:  401 반환     │
```

#### 서버 미들웨어 구현

```typescript
// packages/server/src/firebase/authMiddleware.ts
import { getAuth } from 'firebase-admin/auth'
import type { Request, Response, NextFunction } from 'express'

export interface AuthenticatedRequest extends Request {
  uid: string
  email?: string
}

export async function requireAuth(
  req: AuthenticatedRequest,
  res: Response,
  next: NextFunction
): Promise<void> {
  const authHeader = req.headers.authorization
  if (!authHeader?.startsWith('Bearer ')) {
    res.status(401).json({ error: 'Authorization 헤더가 없습니다.' })
    return
  }

  const idToken = authHeader.split('Bearer ')[1]

  try {
    const decodedToken = await getAuth().verifyIdToken(idToken)
    req.uid = decodedToken.uid
    req.email = decodedToken.email
    next()
  } catch (error) {
    if (error.code === 'auth/id-token-expired') {
      res.status(401).json({ error: '토큰이 만료되었습니다. 재로그인 후 시도하세요.' })
    } else if (error.code === 'auth/invalid-id-token') {
      res.status(401).json({ error: '유효하지 않은 토큰입니다.' })
    } else {
      res.status(401).json({ error: '인증에 실패했습니다.' })
    }
  }
}

// 라우터에 적용 예시
// router.post('/api/projects', requireAuth, createProjectHandler)
```

#### 클라이언트 API 유틸리티

```typescript
// packages/web/src/lib/api.ts
export async function apiRequest<T>(
  path: string,
  options: RequestInit = {}
): Promise<T> {
  const token = await getIdToken()

  const response = await fetch(`${import.meta.env.VITE_API_URL}${path}`, {
    ...options,
    headers: {
      'Content-Type': 'application/json',
      Authorization: `Bearer ${token}`,
      ...options.headers,
    },
  })

  if (response.status === 401) {
    // 토큰 만료: 강제 갱신 후 1회 재시도
    const freshToken = await getIdToken(true)
    const retryResponse = await fetch(
      `${import.meta.env.VITE_API_URL}${path}`,
      {
        ...options,
        headers: {
          'Content-Type': 'application/json',
          Authorization: `Bearer ${freshToken}`,
          ...options.headers,
        },
      }
    )
    if (!retryResponse.ok) throw new ApiError(retryResponse)
    return retryResponse.json()
  }

  if (!response.ok) throw new ApiError(response)
  return response.json()
}
```

## 2. Level 2: Claude API 인증 (BYOK)

### 2.1 개요

BYOK(Bring Your Own Key) 방식으로, 사용자가 자신의 Anthropic API 키를 Agent Council에 등록한다. 키는 서버에서 AES-256-GCM으로 암호화되어 Firestore에 저장되며, Claude Code 세션 시작 시에만 메모리에서 복호화된다.

```
┌─────────────────────────────────────────────────────────────┐
│                   BYOK 인증 흐름                              │
│                                                              │
│  사용자  →  설정 화면  →  API 키 입력                          │
│                               │                              │
│                    서버로 전송 (TLS)                           │
│                               │                              │
│                    서버에서 AES-256-GCM 암호화                 │
│                               │                              │
│                    암호화된 키 → Firestore 저장                │
│                               │                              │
│  Claude Code 세션 시작 시:                                    │
│  Firestore에서 암호화 키 조회                                  │
│     → 서버 환경변수의 마스터 키로 복호화                        │
│     → ANTHROPIC_API_KEY 환경변수로 Claude Code에 주입          │
│     → 사용 후 메모리에서 즉시 삭제                              │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 AES-256-GCM 암호화 상세

#### 암호화 알고리즘 선택 근거

| 알고리즘 | 이유 |
|---|---|
| AES-256 | 256비트 키 — 현재 기준 양자 컴퓨터 위협에도 안전한 수준 |
| GCM 모드 | Galois/Counter Mode — 암호화 + 인증(AEAD)을 동시 제공, 변조 감지 가능 |
| 랜덤 IV | 동일 키로 암호화해도 매번 다른 암호문 생성 |

#### 암호화 구현

```typescript
// packages/server/src/security/encryption.ts
import { createCipheriv, createDecipheriv, randomBytes } from 'crypto'

const ALGORITHM = 'aes-256-gcm'
const KEY_LENGTH = 32    // 256비트
const IV_LENGTH = 12     // GCM 권장 IV 길이 (96비트)
const TAG_LENGTH = 16    // GCM 인증 태그 길이 (128비트)

/**
 * 마스터 암호화 키 — 서버 환경변수에서 로드
 * 형식: BASE64 인코딩된 32바이트 랜덤 값
 * 예시 생성: node -e "console.log(require('crypto').randomBytes(32).toString('base64'))"
 */
function getMasterKey(): Buffer {
  const keyBase64 = process.env.ENCRYPTION_MASTER_KEY
  if (!keyBase64) {
    throw new Error('ENCRYPTION_MASTER_KEY 환경변수가 설정되지 않았습니다.')
  }
  const key = Buffer.from(keyBase64, 'base64')
  if (key.length !== KEY_LENGTH) {
    throw new Error(`암호화 키는 ${KEY_LENGTH}바이트여야 합니다.`)
  }
  return key
}

export interface EncryptedData {
  ciphertext: string   // BASE64
  iv: string          // BASE64 (12바이트 랜덤 IV)
  tag: string         // BASE64 (16바이트 GCM 인증 태그)
}

/**
 * API 키 암호화
 *
 * 구조:
 *   plaintext ──→ AES-256-GCM(key, iv) ──→ { ciphertext, iv, tag }
 *
 * iv는 매 암호화마다 새로 생성 (nonce 재사용 금지)
 */
export function encrypt(plaintext: string): EncryptedData {
  const key = getMasterKey()
  const iv = randomBytes(IV_LENGTH)                    // 12바이트 랜덤 IV

  const cipher = createCipheriv(ALGORITHM, key, iv)
  const encrypted = Buffer.concat([
    cipher.update(plaintext, 'utf8'),
    cipher.final(),
  ])
  const tag = cipher.getAuthTag()                      // GCM 인증 태그

  return {
    ciphertext: encrypted.toString('base64'),
    iv: iv.toString('base64'),
    tag: tag.toString('base64'),
  }
}

/**
 * API 키 복호화
 *
 * GCM 모드는 복호화 시 자동으로 인증 태그를 검증한다.
 * 태그 불일치 시 Error 발생 → 변조된 데이터 감지
 */
export function decrypt(data: EncryptedData): string {
  const key = getMasterKey()
  const iv = Buffer.from(data.iv, 'base64')
  const tag = Buffer.from(data.tag, 'base64')
  const ciphertext = Buffer.from(data.ciphertext, 'base64')

  const decipher = createDecipheriv(ALGORITHM, key, iv)
  decipher.setAuthTag(tag)                             // 인증 태그 설정

  try {
    const decrypted = Buffer.concat([
      decipher.update(ciphertext),
      decipher.final(),                                // 여기서 태그 검증
    ])
    return decrypted.toString('utf8')
  } catch (error) {
    // GCM 인증 실패 — 키 변조 또는 잘못된 마스터 키
    throw new Error('API 키 복호화 실패: 데이터가 변조되었거나 암호화 키가 다릅니다.')
  }
}
```

```typescript
async function rotateEncryptionKey() {
  const oldKey = Buffer.from(process.env.OLD_ENCRYPTION_KEY!, 'base64')
  const newKey = Buffer.from(process.env.ENCRYPTION_MASTER_KEY!, 'base64')

  const usersSnapshot = await db.collection('users')
    .where('authMethod', '==', 'api_key')
    .get()

  const batch = db.batch()
  let count = 0

  for (const userDoc of usersSnapshot.docs) {
    const data = userDoc.data()
    if (!data.apiKeyEncrypted) continue

    // 기존 키로 복호화
    const plaintext = decryptWithKey(data.apiKeyEncrypted, oldKey)

    // 새 키로 재암호화
    const reEncrypted = encryptWithKey(plaintext, newKey)

    batch.update(userDoc.ref, { apiKeyEncrypted: reEncrypted })
    count++

    // 배치 한도 (500) 초과 시 분할
    if (count % 500 === 0) {
      await batch.commit()
    }
  }

  await batch.commit()
  console.log(`키 로테이션 완료: ${count}개 사용자 처리`)
}
```

## 3. Level 3: GitHub API 인증 (PAT)

### 3.1 개요

PR 생성, 브랜치 관리 등 GitHub API 호출에는 Personal Access Token(PAT)이 필요하다. PAT는 Claude API 키와 동일한 AES-256-GCM 암호화 방식으로 Firestore에 저장된다.

PAT는 프로젝트의 Git 설정에 저장되며, GitHub 연동이 필요한 프로젝트에서만 사용된다.

### 3.2 Personal Access Token 등록/저장

#### 등록 UI 흐름

```
[프로젝트 생성 또는 Git 설정 화면]
│
├─ "GitHub 연동" 섹션
│   GitHub Personal Access Token:
│   [••••••••••••••••••••••••••••]  [저장]
│
│   필요한 권한:
│   ✅ repo (프라이빗 레포 접근)
│   ✅ pull_request (PR 생성/조회)
│   ✅ workflow (GitHub Actions 트리거, 선택)
│
│   토큰 생성 방법:
│   GitHub → Settings → Developer Settings → Personal Access Tokens
│   → [Fine-grained tokens 권장]
```

#### PAT 저장 구현

```typescript
// packages/server/src/routes/projects.ts
router.post('/api/projects/:projectId/git/token', requireAuth, async (req, res) => {
  const { projectId } = req.params
  const { githubToken } = req.body

  // 프로젝트 소유자 확인
  const project = await db.collection('projects').doc(projectId).get()
  if (project.data()?.ownerId !== req.uid) {
    return res.status(403).json({ error: '접근 권한이 없습니다.' })
  }

  // PAT 형식 검증
  // Classic PAT: ghp_로 시작
  // Fine-grained PAT: github_pat_로 시작
  if (!githubToken.startsWith('ghp_') && !githubToken.startsWith('github_pat_')) {
    return res.status(400).json({ error: 'GitHub PAT 형식이 올바르지 않습니다.' })
  }

  // GitHub API로 토큰 유효성 및 권한 확인
  const tokenInfo = await validateGitHubToken(githubToken)
  if (!tokenInfo.valid) {
    return res.status(400).json({ error: '유효하지 않은 GitHub 토큰입니다.' })
  }

  // 필요한 스코프 확인
  const requiredScopes = ['repo']
  const missingScopes = requiredScopes.filter(s => !tokenInfo.scopes.includes(s))
  if (missingScopes.length > 0) {
    return res.status(400).json({
      error: `다음 권한이 부족합니다: ${missingScopes.join(', ')}`
    })
  }

  // 암호화 후 저장
  const encryptedToken = encrypt(githubToken)
  await db
    .collection('projects')
    .doc(projectId)
    .collection('git')
    .doc('config')
    .update({
      githubToken: encryptedToken,
      githubTokenUpdatedAt: FieldValue.serverTimestamp(),
      githubLogin: tokenInfo.login,  // 토큰 소유자 GitHub 사용자명 (참고용)
    })

  res.json({ success: true, githubLogin: tokenInfo.login })
})

async function validateGitHubToken(token: string) {
  const response = await fetch('https://api.github.com/user', {
    headers: {
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github+json',
    },
  })

  if (!response.ok) return { valid: false, scopes: [], login: null }

  const scopes = response.headers.get('x-oauth-scopes')?.split(', ') ?? []
  const user = await response.json()
  return { valid: true, scopes, login: user.login }
}
```

### 3.3 필요한 권한 스코프

#### Classic PAT 스코프

| 스코프 | 필수 여부 | 용도 |
|---|---|---|
| `repo` | 필수 | 프라이빗 레포 전체 접근 (클론, 푸시, PR) |
| `repo:status` | `repo`에 포함 | 커밋 상태 업데이트 |
| `repo:deployment` | `repo`에 포함 | 배포 관련 |
| `public_repo` | 퍼블릭 레포만 | 퍼블릭 레포 전용 시 더 좁은 권한 |
| `workflow` | 선택 | GitHub Actions 트리거 (CI 연동 시) |
| `read:org` | 선택 | 조직 레포 목록 조회 |

#### Fine-grained PAT 권한 (권장)

Fine-grained PAT는 레포지토리별, 권한별로 세밀하게 설정할 수 있어 최소 권한 원칙에 부합한다.

| 권한 항목 | 수준 | 용도 |
|---|---|---|
| Contents | Read & Write | 코드 읽기/쓰기, 브랜치 생성 |
| Pull requests | Read & Write | PR 생성, 댓글, 병합 |
| Commit statuses | Read & Write | CI 상태 표시 |
| Issues | Read (선택) | 이슈 참조 |
| Workflows | Read & Write (선택) | Actions 트리거 |

#### PAT를 사용하는 Council Server 기능

```typescript
// packages/server/src/git/githubClient.ts
export async function createPullRequest(
  projectId: string,
  params: {
    title: string
    body: string
    sourceBranch: string
    targetBranch: string
    repoFullName: string  // "owner/repo"
  }
): Promise<{ number: number; url: string }> {
  // 프로젝트 Git 설정에서 PAT 복호화
  const gitConfig = await db
    .collection('projects').doc(projectId)
    .collection('git').doc('config')
    .get()

  const encryptedToken = gitConfig.data()?.githubToken
  if (!encryptedToken) {
    throw new Error('GitHub 토큰이 설정되지 않았습니다.')
  }

  const token = decrypt(encryptedToken)  // 메모리에서만 복호화

  const response = await fetch(
    `https://api.github.com/repos/${params.repoFullName}/pulls`,
    {
      method: 'POST',
      headers: {
        Authorization: `Bearer ${token}`,
        Accept: 'application/vnd.github+json',
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({
        title: params.title,
        body: params.body,
        head: params.sourceBranch,
        base: params.targetBranch,
      }),
    }
  )

  // token 변수는 이 함수 스코프에서만 존재 — 함수 반환 시 GC 대상
  const pr = await response.json()
  return { number: pr.number, url: pr.html_url }
}
```

```typescript
  color: z.string().regex(/^#[0-9a-fA-F]{6}$/, '유효한 색상 코드여야 합니다.'),
})

// 메시지 스키마
export const SendMessageSchema = z.object({
  content: z
    .string()
    .min(1)
    .max(10000, '메시지는 10,000자 이하여야 합니다.'),
  replyTo: z.string().optional(),
})

// 검증 미들웨어
export function validate<T>(schema: z.ZodSchema<T>) {
  return (req: Request, res: Response, next: NextFunction) => {
    const result = schema.safeParse(req.body)
    if (!result.success) {
      const errors = result.error.errors.map(e => ({
        field: e.path.join('.'),
        message: e.message,
      }))
      return res.status(400).json({ error: '입력값이 올바르지 않습니다.', errors })
    }
    req.body = result.data  // 검증된 데이터로 교체
    next()
  }
}
```

#### CSRF 방지

Agent Council은 **Bearer Token 방식**을 사용하므로 전통적인 CSRF 공격에 구조적으로 안전하다.

```
CSRF 방지 이유:
────────────────────────────────────────────────────────
1. 인증 방식: Authorization: Bearer <token> 헤더 사용
   → 쿠키 기반 인증이 아니므로 CSRF 공격 벡터 없음
   → 악의적 사이트가 다른 도메인에서 Bearer 헤더를
      자동으로 추가할 수 없음 (Same-Origin Policy)

2. Content-Type: application/json 요청만 수락
   → HTML form의 자동 제출(CSRF 공격 방식)은
      application/x-www-form-urlencoded 또는
      multipart/form-data 사용 → 서버에서 거부

3. CORS 설정으로 허가된 오리진만 요청 가능
────────────────────────────────────────────────────────
```

```typescript
// packages/server/src/middleware/security.ts
// JSON Content-Type 강제 (CSRF 추가 방어)
app.use('/api/', (req, res, next) => {
  if (['POST', 'PUT', 'PATCH'].includes(req.method)) {
    const contentType = req.headers['content-type']
    if (!contentType?.includes('application/json') &&
        !contentType?.includes('multipart/form-data')) {
      return res.status(415).json({ error: '지원하지 않는 Content-Type입니다.' })
    }
  }
  next()
})
```

## 5. 데이터 보안

### 5.1 Firestore Security Rules 요약

> 상세 Rules 및 테스트는 `/firebase/firestore.rules` 및 `07_운영/보안설정.md` 참조

핵심 원칙:

```
┌──────────────────────────────────────────────────────────────┐
│               Firestore Security Rules 원칙                   │
│                                                               │
│  1. 인증 필수: 모든 데이터 접근은 로그인 사용자만               │
│     request.auth != null                                      │
│                                                               │
│  2. 소유자 격리: 사용자 문서는 본인만 읽기/쓰기                  │
│     request.auth.uid == userId                               │
│                                                               │
│  3. 프로젝트 소유자 검증: 프로젝트 하위 자원 접근 시             │
│     projects/{id}.ownerId == request.auth.uid                │
│                                                               │
│  4. 민감 필드 보호: API 키 필드는 서버 SDK로만 접근             │
│     (아래 5.2 참조)                                           │
│                                                               │
│  5. 최소 권한: 읽기/쓰기 권한을 필요한 범위만 부여              │
└──────────────────────────────────────────────────────────────┘
```

```javascript
// firebase/firestore.rules (핵심 부분)
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // 헬퍼 함수
    function isAuthenticated() {
      return request.auth != null;
    }

    function isOwner(userId) {
      return request.auth.uid == userId;
    }

    function isProjectOwner(projectId) {
      return isAuthenticated() &&
        get(/databases/$(database)/documents/projects/$(projectId))
          .data.ownerId == request.auth.uid;
    }

    // 사용자 문서: 본인만 접근
    match /users/{userId} {
      allow read, write: if isAuthenticated() && isOwner(userId);
    }

    // 프로젝트: 소유자만 접근
    match /projects/{projectId} {
      allow read, write: if isAuthenticated()
        && resource.data.ownerId == request.auth.uid;

      // 하위 컬렉션은 프로젝트 소유자만
      match /{subcollection=**} {
        allow read, write: if isProjectOwner(projectId);
      }
    }
  }
}
```

### 5.2 API 키 필드 클라이언트 노출 방지

#### 문제

Firestore Security Rules가 `users/{userId}` 전체 문서 읽기를 허용하면, 클라이언트가 암호화된 API 키 데이터(`apiKeyEncrypted`)를 받을 수 있다.

암호화는 되어 있지만, 암호문이 클라이언트에 노출되는 것도 불필요하다.

#### 해결책 1: 서버 API 경유 (권장)

클라이언트가 민감한 사용자 설정 데이터는 Firestore 직접 접근 대신 Council Server API를 통해 조회한다.

```typescript
// packages/server/src/routes/settings.ts
router.get('/api/settings', requireAuth, async (req, res) => {
  const userDoc = await db.collection('users').doc(req.uid).get()
  const data = userDoc.data()

  // 클라이언트에 반환 시 민감 필드 제거
  const { apiKeyEncrypted, maxPlanTokenPath, ...safeData } = data ?? {}

  res.json({
    ...safeData,
    hasApiKey: !!apiKeyEncrypted,           // 키 등록 여부만 전달
    hasMaxPlanToken: !!maxPlanTokenPath,    // 토큰 등록 여부만 전달
    // 실제 키 내용은 절대 반환하지 않음
  })
})
```

#### 해결책 2: Firestore Field Mask (보조)

클라이언트에서 Firestore를 직접 읽을 때도 특정 필드만 요청할 수 있다.

```typescript
// packages/web/src/firebase/user.ts
import { getDoc, doc } from 'firebase/firestore'

// 민감 필드를 제외한 사용자 정보만 조회
// 단, Firestore Security Rules 레벨에서는 필드별 제어가 불가하므로
```

### 5.3 로그에서 민감정보 마스킹

Council Server의 모든 로그에서 API 키, 토큰 등 민감정보를 자동으로 마스킹한다.

```typescript
// packages/server/src/utils/logger.ts
import pino from 'pino'

const SENSITIVE_PATTERNS: Array<{ pattern: RegExp; replacement: string }> = [
  // Anthropic API 키
  { pattern: /sk-ant-[a-zA-Z0-9_-]+/g, replacement: 'sk-ant-***MASKED***' },
  // GitHub PAT (Classic)
  { pattern: /ghp_[a-zA-Z0-9]+/g, replacement: 'ghp_***MASKED***' },
  // GitHub PAT (Fine-grained)
  { pattern: /github_pat_[a-zA-Z0-9_]+/g, replacement: 'github_pat_***MASKED***' },
  // Firebase ID Token (JWT)
  { pattern: /eyJ[a-zA-Z0-9_-]+\.[a-zA-Z0-9_-]+\.[a-zA-Z0-9_-]+/g, replacement: 'JWT_***MASKED***' },
  // 암호화된 데이터 필드
  { pattern: /"(apiKeyEncrypted|githubToken|maxPlanTokenPath)":\s*"[^"]+"/g,
    replacement: '"$1": "***MASKED***"' },
]

function maskSensitiveData(message: string): string {
  let masked = message
  for (const { pattern, replacement } of SENSITIVE_PATTERNS) {
    masked = masked.replace(pattern, replacement)
  }
  return masked
}

export const logger = pino({
  serializers: {
    // 요청 로그에서 Authorization 헤더 마스킹
    req: (req) => ({
      method: req.method,
      url: req.url,
      headers: {
        ...req.headers,
        authorization: req.headers.authorization ? 'Bearer ***MASKED***' : undefined,
      },
    }),
    err: (err) => ({
      type: err.constructor.name,
      message: maskSensitiveData(err.message),
      stack: maskSensitiveData(err.stack),
    }),
  },
  hooks: {
    logMethod(inputArgs, method) {
      // 모든 로그 메시지 마스킹
      if (typeof inputArgs[0] === 'string') {
        inputArgs[0] = maskSensitiveData(inputArgs[0])
      }
      method.apply(this, inputArgs)
    },
  },
})
```
