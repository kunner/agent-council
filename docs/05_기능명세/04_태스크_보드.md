## 1. 개념 및 역할

### 1.1 태스크 보드란

태스크 보드는 **프로젝트에 종속된 칸반 보드**다. 하나의 프로젝트는 정확히 하나의 태스크 보드를 가지며, Council Room에서 합의된 작업 항목(태스크)을 시각적으로 추적·관리하는 도구다.

```
Project
├── Git Repository
├── Council Room     ← 대화/의사결정
├── Workspaces       ← Set별 worktree
├── Board            ← 태스크 보드 (이 문서의 대상)
└── Artifacts        ← PR, 빌드 결과
```

### 1.2 보드와 Council Room의 관계

태스크 보드는 Council Room 대화의 **결과물 저장소**이자 **진행 현황판**이다.

```
Council Room (대화)          태스크 보드 (추적)
─────────────────            ─────────────────
리더 합의 → 태스크 생성  →→→  Backlog 카드 추가
Set 할당 결정           →→→  카드에 Set 배지 표시
코딩 완료 보고          →→→  In Progress → Review 이동
PR 머지 완료           →→→  Review → Done 이동
버그 발견              →→→  Backlog에 신규 카드 생성
```

### 1.3 보드가 없으면 생기는 문제

- "어떤 작업이 누구에게 할당됐는지" 대화를 다시 뒤져야 함
- 병렬 Set들 간 작업 의존 관계를 관리하기 어려움
- PM이 진행 상황을 한눈에 파악 불가
- 브랜치, PR과 작업 항목의 연결고리가 없어 맥락 손실 발생

## 2. 칸반 열 정의

보드는 5개 열로 구성된다. 기본 흐름은 왼쪽에서 오른쪽이며, **Blocked**는 어느 열에서든 진입 가능한 특수 상태다.

```
┌──────────┬─────────────┬──────────┬──────────┐    ┌──────────┐
│ Backlog  │ In Progress │  Review  │   Done   │    │ Blocked  │
│          │             │          │          │    │          │
│ 대기 중인 │  현재 작업  │  완료 후  │  검증    │    │  진행    │
│ 모든 작업 │     중      │  리뷰 단계│  완료    │    │  중단    │
└──────────┴─────────────┴──────────┴──────────┘    └──────────┘
```

### 2.1 Backlog

- **의미**: 아직 시작되지 않은 작업 항목 전체
- **진입 조건**: 태스크 생성 시 기본 배치 (상태값: `backlog`)
- **이탈 조건**: Set이 작업을 시작하면 In Progress로 이동
- **표시 정보**: 제목, 우선순위 배지, 담당 Set (미할당 가능), 의존성 아이콘
- **특이사항**: 우선순위순 + 생성일순으로 자동 정렬. 미할당 태스크도 존재 가능

### 2.2 In Progress

- **의미**: 담당 Set이 현재 작업 중인 항목
- **진입 조건**: Set 리더 또는 PM이 `backlog → in_progress` 전환을 수행
- **이탈 조건**:
  - 작업 완료 → Review로 이동
  - 블로커 발생 → Blocked로 이동
- **표시 정보**: 담당 Set 색상 배지, 연결된 브랜치명, 진행률(선택)
- **특이사항**: 선행 태스크(dependency)가 완료되지 않은 상태에서는 In Progress 전환 시 경고 표시

### 2.3 Review

- **의미**: 구현 완료 후 다른 Set 또는 PM의 검토를 기다리는 항목
- **진입 조건**: 담당 Set 리더가 "완료" 보고 시 자동 또는 수동 전환
- **이탈 조건**:
  - 리뷰 통과 + PR 머지 완료 → Done으로 이동
  - 리뷰에서 수정 요청 → In Progress로 복귀
- **표시 정보**: 연결된 PR 링크, 리뷰 요청 Set, 생성된 PR 번호
- **특이사항**: PR이 연결된 경우 PR 상태(open/approved/merged)를 실시간 반영

### 2.4 Done

- **의미**: PR 머지까지 완료된 항목 (또는 PM이 완료 처리한 항목)
- **진입 조건**:
  - PR 머지 이벤트 수신 시 Council Server가 자동 전환
  - PM이 수동으로 Done 처리
- **이탈 조건**: 없음 (재오픈 필요 시 PM이 수동으로 Backlog 복귀)
- **표시 정보**: 완료 일시, 머지된 커밋 해시, 담당 Set
- **특이사항**: 기본적으로 접혀있음 (토글로 펼치기). 최근 완료 5개만 기본 노출

### 2.5 Blocked

- **의미**: 외부 요인으로 진행이 중단된 항목
- **진입 조건**: 어느 열에서든 전환 가능. 전환 시 블로킹 사유 입력 필수
- **이탈 조건**: 블로커 해소 후 원래 열로 복귀 (사유 입력 필수)
- **표시 정보**: 블로킹 사유, 블로킹 시작 시각, 블로킹 관련 Council 메시지 링크
- **특이사항**: Blocked 카드는 원래 열 위치를 기억해 복귀 시 해당 열로 돌아감. Council Room에 자동 알림 발송

## 3. 태스크 데이터 모델

PLAN.md 섹션 3.5 및 Firestore 스키마(섹션 6.1)를 기반으로 한 전체 타입 정의:

```typescript
// packages/shared/src/types/task.ts

type TaskStatus = 'backlog' | 'in_progress' | 'review' | 'done' | 'blocked'
type TaskPriority = 'critical' | 'high' | 'medium' | 'low'

interface Task {
  // 식별자
  id: string
  projectId: string

  // 내용
  title: string                        // 최대 100자
  description: string                  // 마크다운 지원, 최대 5000자

  // 상태
  status: TaskStatus
  blockedReason?: string               // status === 'blocked' 일 때 필수
  blockedFromStatus?: TaskStatus       // blocked 이전 상태 (복귀 대상)

  // 할당
  assignedSetId?: string               // 미할당 허용

  // 우선순위
  priority: TaskPriority

  // 의존성
  dependencies: string[]               // 선행 태스크 ID 목록
  blockedBy: string[]                  // 이 태스크를 블로킹하는 태스크 ID (역방향 참조)

  // Git 연동
  branch?: string                      // 예: "set-b/backend"
  pullRequestUrl?: string              // GitHub PR URL
  pullRequestId?: string               // Firestore pullRequests/{prId}
  commitHash?: string                  // 완료 시 머지 커밋 해시

  // Council 연동
  createdFromMessageId?: string        // 생성 출처 메시지 ID
  relatedMessageIds: string[]          // 이 태스크와 연관된 메시지 ID 목록

  // 생성 방식 추적
  creationMethod: 'council_extract' | 'pm_direct' | 'leader_subtask'
  parentTaskId?: string                // 서브태스크인 경우 부모 태스크 ID

  // 서브태스크
  subTaskIds: string[]                 // 이 태스크에서 분해된 서브태스크 ID 목록

  // 메타
  createdAt: Timestamp
  updatedAt: Timestamp
  completedAt?: Timestamp
}
```

**Firestore 경로**: `projects/{projectId}/tasks/{taskId}`

## 4. 태스크 CRUD 상세

### 4.1 생성 (Create)

**API**: `POST /api/projects/{projectId}/tasks`

**요청 바디**:
```typescript
{
  title: string                 // 필수
  description?: string          // 선택, 기본값 ""
  priority?: TaskPriority       // 선택, 기본값 "medium"
  assignedSetId?: string        // 선택
  dependencies?: string[]       // 선택, 기본값 []
  createdFromMessageId?: string // 선택
  creationMethod: 'council_extract' | 'pm_direct' | 'leader_subtask'
  parentTaskId?: string         // leader_subtask인 경우
}
```

**처리 흐름**:
```
1. 입력값 검증 (title 필수, 길이 제한 등)
2. Firestore tasks 컬렉션에 문서 생성
   - status: 'backlog' (항상 backlog에서 시작)
   - id: nanoid() 생성
3. 의존성 역방향 참조 업데이트
   - dependencies 배열의 각 태스크에 blockedBy 필드 업데이트
4. 서브태스크인 경우 부모 태스크 subTaskIds 업데이트
5. Council Room에 시스템 알림 발송 (createdFromMessageId 있는 경우)
6. 생성된 태스크 반환
```

**응답**: 생성된 `Task` 객체

### 4.2 조회 (Read)

#### 4.2.1 목록 조회

**API**: `GET /api/projects/{projectId}/tasks`

**쿼리 파라미터**:
```
status?: TaskStatus | 'all'     기본값: 'all'
assignedSetId?: string
priority?: TaskPriority
sortBy?: 'priority' | 'created' | 'updated'
sortDir?: 'asc' | 'desc'
```

**Firestore 쿼리**:
```typescript
const tasksRef = collection(db, `projects/${projectId}/tasks`)
let q = query(tasksRef, orderBy('priority'), orderBy('createdAt'))

if (status !== 'all') {
  q = query(q, where('status', '==', status))
}
if (assignedSetId) {
  q = query(q, where('assignedSetId', '==', assignedSetId))
}
```

#### 4.2.2 단건 조회

**API**: `GET /api/projects/{projectId}/tasks/{taskId}`

**응답**: `Task` 객체 + 연관 메시지 요약 + 연관 PR 정보

#### 4.2.3 실시간 구독 (클라이언트)

```typescript
// 보드 UI에서 태스크 실시간 구독
const tasksRef = collection(db, `projects/${projectId}/tasks`)
const q = query(tasksRef, orderBy('updatedAt', 'desc'))

onSnapshot(q, (snapshot) => {
  snapshot.docChanges().forEach((change) => {
    if (change.type === 'added')    addTaskToBoard(change.doc.data())
    if (change.type === 'modified') updateTaskOnBoard(change.doc.data())
    if (change.type === 'removed')  removeTaskFromBoard(change.doc.id)
  })
})
```

### 4.3 수정 (Update)

**API**: `PATCH /api/projects/{projectId}/tasks/{taskId}`

**수정 가능 필드**:
```typescript
{
  title?: string
  description?: string
  status?: TaskStatus
  blockedReason?: string       // status를 'blocked'로 변경할 때 필수
  assignedSetId?: string | null
  priority?: TaskPriority
  dependencies?: string[]
  branch?: string
  pullRequestUrl?: string
}
```

**상태 변경 시 처리**:
- `blocked`로 변경: `blockedFromStatus` 자동 기록, `blockedReason` 필수 검증
- `blocked`에서 이탈: `blockedFromStatus`로 복귀, `blockedReason` 초기화
- `done`으로 변경: `completedAt` 자동 기록
- 의존성 변경 시: 역방향 참조(`blockedBy`) 자동 재계산

**낙관적 업데이트 (클라이언트)**:
```
사용자 드래그앤드롭 또는 상태 변경
  → UI 즉시 반영 (낙관적)
  → 서버로 PATCH 요청
  → 성공: Firestore onSnapshot으로 자연스럽게 동기화
  → 실패: 오류 토스트 + UI 원상 복구
```

### 4.4 삭제 (Delete)

**API**: `DELETE /api/projects/{projectId}/tasks/{taskId}`

**삭제 전 검증**:
- `in_progress` 또는 `review` 상태인 경우 확인 모달 표시
- 다른 태스크의 `dependencies`에 참조된 경우 경고 표시 (삭제는 허용하되 역방향 참조 정리)

**처리 흐름**:
```
1. status 확인 → 활성 상태면 확인 모달
2. 이 태스크를 dependencies에 포함한 다른 태스크의 dependencies 배열 정리
3. 이 태스크를 blockedBy에 포함한 다른 태스크의 blockedBy 배열 정리
4. 부모 태스크의 subTaskIds에서 제거
5. Firestore 문서 삭제
6. Council Room에 시스템 알림 (선택)
```

## 5. 태스크 생성 방식 3가지

### 5.1 방식 1 — Council 대화에서 자동 추출

**트리거**: PM 또는 리더가 Council Room 메시지에서 "태스크로 등록해" 또는 유사 명령 입력

**상세 플로우**:

```
[Council Room 대화]

  🎯 아키텍처팀: DB 스키마부터 만들어야 합니다.
                 User, Room, Message 테이블이 핵심입니다.

  👤 PM: 좋아, 태스크로 등록해
         ↓
[Council Server 처리]
  1. 메시지 파싱: "태스크로 등록해" 패턴 감지
  2. 직전 컨텍스트 메시지 분석 (LLM 호출)
     → 제목 추출: "DB 스키마 설계"
     → 담당 Set 추론: 아키텍처팀 언급됨 → Set A
     → 우선순위 추론: 선행 작업으로 언급 → high
  3. Firestore tasks에 생성
     - creationMethod: 'council_extract'
     - createdFromMessageId: <PM 메시지 ID>
  4. Council Room에 인라인 카드로 확인 메시지 발송

[Council Room 결과]

  ⚙️ 시스템: 태스크가 생성되었습니다.
  ┌─────────────────────────────┐
  │ 📋 DB 스키마 설계            │
  │ 🎯 아키텍처팀 | 🔴 High      │
  │ 상태: Backlog                │
  │ [보드에서 보기] [수정]        │
  └─────────────────────────────┘
```

**파싱 트리거 패턴** (대소문자/공백 무관):
- "태스크로 등록해"
- "태스크 만들어"
- "태스크 추가해"
- "보드에 올려"
- "보드에 등록해"
- "작업으로 등록"
- "이거 태스크로"

**컨텍스트 추출 규칙**:
- 직전 10개 메시지를 컨텍스트로 사용
- LLM이 제목, 담당 Set, 우선순위를 추론
- 추론 결과는 확인 카드로 노출, PM이 수정 가능

### 5.2 방식 2 — PM이 보드 UI에서 직접 생성

**진입점**: 보드 화면 우상단 "+ 태스크 추가" 버튼 또는 각 열 상단의 "+" 아이콘

**UI 인터랙션 플로우**:

```
[태스크 생성 인터랙션]

1. "+ 태스크 추가" 클릭
   → 슬라이드인 패널 열림 (오른쪽 사이드패널)

2. 패널 내 폼:
   ┌─────────────────────────────────────┐
   │ 새 태스크                      [✕]  │
   ├─────────────────────────────────────┤
   │ 제목 *                              │
   │ ┌─────────────────────────────────┐ │
   │ │ 태스크 제목 입력...              │ │
   │ └─────────────────────────────────┘ │
   │                                     │
   │ 설명 (마크다운 지원)                 │
   │ ┌─────────────────────────────────┐ │
   │ │                                 │ │
   │ │                                 │ │
   │ └─────────────────────────────────┘ │
   │                                     │
   │ 우선순위          담당 Set           │
   │ [Medium ▼]        [미할당 ▼]        │
   │                                     │
   │ 선행 태스크 (의존성)                 │
   │ [+ 의존성 추가]                      │
   │                                     │
   │ Council 메시지 연결                  │
   │ [메시지 검색...]                     │
   │                                     │
   │         [취소]  [태스크 생성]        │
   └─────────────────────────────────────┘

3. "태스크 생성" 클릭
   → creationMethod: 'pm_direct'
   → Firestore 저장
   → 패널 닫힘
   → 보드에 카드 즉시 나타남 (낙관적 업데이트)
```

**열 상단 "+" 아이콘 동작**:
- Backlog 열의 "+"를 클릭하면 status: 'backlog'로 미리 설정된 패널 열림
- In Progress 열의 "+"는 assignedSetId 필드 필수로 표시

## 6. 태스크 상태 전이 규칙

### 6.1 상태 다이어그램

```
              ┌──────────────────────────────────────┐
              │              생성                     │
              ▼                                      │
         ┌─────────┐                                 │
    ┌──→ │ backlog │ ←──────────────────────────┐   │
    │    └────┬────┘                            │   │
    │         │ Set 할당 후 작업 시작             │   │
    │         │ (리더 또는 PM)                   │   │
    │         ▼                                 │   │
    │    ┌─────────────┐  리뷰 수정 요청          │   │
    │    │ in_progress │ ←─────────────┐        │   │
    │    └──────┬──────┘               │        │   │
    │           │ 작업 완료              │        │   │
    │           │ (리더 "완료" 보고)     │        │   │
    │           ▼                      │        │   │
    │      ┌────────┐                  │        │   │
    │      │ review │ ─────────────────┘        │   │
    │      └───┬────┘ 수정 요청                  │   │
    │          │ PR 머지 완료                    │   │
    │          │ (자동 또는 PM)                  │   │
    │          ▼                                │   │
    │       ┌──────┐                           │   │
    │       │ done │ ────────────── PM 재오픈 ──┘   │
    │       └──────┘                               │
    │                                              │
    │    ┌─────────┐                               │
    └─── │ blocked │ ◄─────────── 어느 상태에서든  │
         └────┬────┘   블로커 발생 시              │
              │                                    │
              └── 블로커 해소 → 이전 상태로 복귀 ───┘
```

### 6.2 허용된 전이표

| 현재 상태 → | backlog | in_progress | review | done | blocked |
|---|---|---|---|---|---|
| **backlog** | — | O | X | X | O |
| **in_progress** | X | — | O | X | O |
| **review** | X | O (수정요청) | — | O | O |
| **done** | O (재오픈) | X | X | — | X |
| **blocked** | O | O | O | X | — |

- **X**: 해당 전이 불가 (API에서 400 반환)
- **O**: 허용
- **done → blocked**: 완료된 작업은 blocked 불가 (재오픈 후 blocked 처리)

### 6.3 각 전이의 선행 조건

**backlog → in_progress**
- 조건: `assignedSetId`가 설정되어 있을 것
- 조건: 모든 `dependencies` 태스크가 `done` 상태일 것
  - 미충족 시: 경고 표시 후 PM 확인 요청 (강제 진행 허용)
- 자동화: Set 리더가 "작업 시작합니다" 메시지 발송 시 Council Server가 자동 전환

**in_progress → review**
- 조건: `branch`가 설정되어 있을 것 (코드 작업인 경우)
- 자동화: PR 생성 이벤트 수신 시 Council Server가 자동 전환

**review → done**
- 자동화: PR 머지 이벤트 수신 시 Council Server가 `completedAt` 기록과 함께 자동 전환
- 수동: PM이 직접 Done으로 이동 (PR 없는 태스크, 예: 문서 작성)

**review → in_progress**
- 조건: 리뷰어가 수정 요청 의견을 남겼을 것
- 트리거: Council Room에서 "수정 요청" 메시지 또는 보드 UI에서 수동 전환

**any → blocked**
- 조건: `blockedReason` 입력 필수 (최소 10자)
- 처리: `blockedFromStatus` 현재 상태로 기록
- 알림: Council Room에 자동 시스템 알림 발송

**blocked → any**
- 조건: 복귀 사유 입력 필수
- 기본 복귀 대상: `blockedFromStatus` 값
- 알림: Council Room에 자동 시스템 알림 발송

### 6.4 자동 전이 트리거 (Council Server)

```typescript
// Council Server의 Git 이벤트 핸들러
async function handleGitEvent(event: GitEvent, projectId: string) {
  switch (event.type) {

    case 'pr_opened': {
      // PR이 열리면 해당 브랜치와 연결된 태스크를 review로 전환
      const task = await findTaskByBranch(projectId, event.branch)
      if (task && task.status === 'in_progress') {
        await updateTaskStatus(task.id, 'review', {
          pullRequestUrl: event.prUrl,
          pullRequestId: event.prId,
        })
        await sendCouncilMessage(projectId, {
          type: 'system',
          content: `태스크 "${task.title}"가 Review 단계로 이동했습니다.\nPR: ${event.prUrl}`,
          metadata: { taskRefs: [task.id], pullRequestUrl: event.prUrl },
        })
      }
      break
    }

    case 'pr_merged': {
      // PR 머지 시 태스크를 done으로 전환
      const task = await findTaskByPrUrl(projectId, event.prUrl)
      if (task && task.status === 'review') {
        await updateTaskStatus(task.id, 'done', {
          commitHash: event.mergeCommit,
          completedAt: Timestamp.now(),
        })
      }
      break
    }
  }
}
```

## 8. 우선순위 관리

### 8.1 우선순위 단계

| 레벨 | 표시 | 색상 | 의미 |
|---|---|---|---|
| **critical** | 🔴 Critical | red-500 | 프로젝트 차단 수준. 즉시 처리 필요 |
| **high** | 🟠 High | orange-500 | 현재 스프린트 내 처리 필요 |
| **medium** | 🟡 Medium | yellow-500 | 일반 작업. 기본값 |
| **low** | 🩵 Low | blue-300 | 시간 여유 있을 때 처리 |

### 8.2 자동 우선순위 추론

Council 대화에서 태스크 추출 시 LLM이 우선순위 추론:

```
추론 근거:
- "지금 당장", "즉시", "블로킹" 키워드 → critical
- "이번 주", "먼저", "선행" 키워드 → high

```

### 8.3 보드에서의 정렬

- 같은 열 내에서 우선순위 순으로 자동 정렬 (critical → high → medium → low)
- 동일 우선순위 내에서는 생성일 오름차순
- PM이 드래그앤드롭으로 수동 재정렬 시 `manualOrder` 필드로 위치 고정

### 8.4 Critical 태스크 특수 처리

```
critical 태스크 생성 시:
1. 보드 해당 열 맨 위에 고정 표시 (붉은 테두리)
2. Council Room에 즉시 알림 발송
   "@all [CRITICAL] 태스크 [제목] 등록됨"
3. 담당 Set 리더에게 멘션
4. 미할당 상태라면 PM에게 즉시 할당 요청

critical 태스크가 blocked 전환 시:
1. Council Room에 긴급 알림
2. 모든 리더에게 멘션
```

## 9. 의존성 관리

### 9.1 의존성 개념

- 태스크 A가 태스크 B를 `dependencies`에 포함하면 → "A는 B가 완료된 후 시작 가능"
- B는 A를 `blockedBy`에 포함 (역방향 참조, 자동 관리)

```
태스크 의존성 예시:

  [DB 스키마 설계]  ──→  [백엔드 API 구현]  ──→  [프론트 API 연동]
       (A)                    (B)                      (C)
                    A 완료 후 B 시작 가능     B 완료 후 C 시작 가능
```

### 9.2 의존성 시각화

```
보드 카드 내 의존성 표시:

  ┌─────────────────────────┐
  │ 백엔드 API 구현           │
  │ 🟢 백엔드팀 | 🔴 High    │
  │                         │
  │ ⛓ 선행: DB 스키마 설계   │  ← done 상태면 녹색, 아니면 노란색
  │ ⛓ 후행: 프론트 API 연동  │  ← 이 태스크가 완료돼야 후행 태스크 시작
  └─────────────────────────┘
```

### 9.3 의존성 위반 처리

**backlog → in_progress 전환 시 미완료 선행 태스크 존재**:
```
경고 모달:

  ⚠️ 선행 태스크가 완료되지 않았습니다

  다음 태스크가 아직 완료되지 않았습니다:
  - [DB 스키마 설계] (in_progress)

  그래도 작업을 시작하시겠습니까?

  [취소]  [강제 시작]
```

**순환 의존성 감지**:
```
의존성 추가 시 서버에서 DFS로 순환 검사:
  - 순환 탐지 시 400 Bad Request 반환
  - 오류 메시지: "순환 의존성이 발생합니다: A → B → C → A"
```

### 9.4 의존성 완료 자동 알림

```
선행 태스크 A가 done으로 전환될 때:
  → A를 dependencies에 포함한 모든 태스크 조회
  → 해당 태스크의 모든 선행 태스크가 done인지 확인
  → 전부 done이면:
     - 담당 Set 리더에게 알림: "태스크 [B] 시작 가능 상태가 되었습니다"
     - 보드에서 해당 태스크 카드 강조 표시 (1분간 pulse 애니메이션)
```

## 10. Git 연동

### 10.1 브랜치 자동 연결

태스크가 `in_progress`로 전환될 때 Set의 브랜치를 자동으로 연결:

```
1. 태스크가 assignedSetId: 'set-b'로 in_progress 전환
2. Council Server가 Set B의 현재 브랜치 조회
   → sets/{setId}.branch: "set-b/backend"
3. 태스크의 branch 필드 자동 업데이트
4. 보드 카드에 브랜치 배지 표시: [🔀 set-b/backend]
```

**브랜치 수동 연결**:
- 태스크 상세 패널에서 브랜치명 직접 입력 가능
- 입력 시 해당 프로젝트의 브랜치 목록으로 자동완성 제공

### 10.2 PR 자동 연결

```
Set 리더가 "작업 완료. PR 올립니다." 메시지 발송
  → Council Server가 GitHub API로 PR 생성
  → PR 정보를 Firestore pullRequests 컬렉션에 저장
  → 동일 branch를 가진 태스크 조회
  → 태스크의 pullRequestUrl, pullRequestId 업데이트
  → 태스크 상태 in_progress → review 자동 전환
  → 보드 카드에 PR 배지 표시: [PR #42]
```

### 10.3 커밋 해시 기록

```
PR 머지 완료 이벤트:
  → 태스크 status: 'done'
  → commitHash: "abc1234" (머지 커밋)
  → completedAt: 현재 시각
```

### 10.4 보드 카드의 Git 상태 표시

```
┌─────────────────────────────┐
│ 백엔드 API 구현               │
│ 🟢 백엔드팀 | 🔴 High        │
│                             │
│ 🔀 set-b/backend            │  ← 연결된 브랜치
│ #42 open ▲423 ▼12          │  ← PR 번호 + diff 요약
└─────────────────────────────┘
```

### 11.1 생성 출처 추적

```
태스크 상세 패널 하단:

  📎 생성 출처
  ┌──────────────────────────────────────────┐

  │ "좋아, 태스크로 등록해"                    │

  │                                   [이동→] │
  └──────────────────────────────────────────┘
```

### 11.2 연관 메시지 자동 수집

태스크가 생성된 이후 Council Room 메시지에서 태스크 ID가 언급되거나 태스크 제목과 유사한 내용이 감지될 때 `relatedMessageIds`에 자동 추가:

```
자동 연결 조건:
  - 시스템 메시지의 metadata.taskRefs에 태스크 ID 포함
  - 리더 메시지에서 태스크 제목 직접 언급 (문자열 매칭)
```

### 11.3 태스크 → Council Room 딥링크

보드 카드의 "[Council에서 보기]" 버튼 클릭 시:
- Council Room으로 이동
- `createdFromMessageId` 메시지 위치로 자동 스크롤
- 해당 메시지 하이라이트 (3초 pulse)

### 11.4 Council Room 인라인 카드

Council Room 메시지 스트림에서 태스크 관련 이벤트는 인라인 카드로 표시:

```
생성 시:
  ┌─ 📋 태스크 생성됨 ───────────────────┐
  │  DB 스키마 설계                       │
```

## 12. 드래그앤드롭 동작

라이브러리: `@hello-pangea/dnd` (PLAN.md 4.6 UI 기술 선택)

### 12.1 기본 동작

```
[드래그 시작]
  - 카드 집어들기: 마우스 버튼 누름 + 100ms 이상 유지
  - 카드가 살짝 커지며 그림자 생김 (scale 1.02, shadow-lg)
  - 드래그 중인 카드 자리에 반투명 플레이스홀더 표시

[드래그 중]
  - 드래그 중인 카드가 마우스 커서를 따라 이동
  - 유효한 드롭 영역(다른 열) 위에 있을 때 열 배경색 강조
  - 무효한 영역(보드 외부) 위에 있을 때 금지 커서 표시

[드롭]
  - 유효한 열에 드롭: 낙관적 업데이트 → 서버 PATCH 요청
  - 같은 열 내 드롭: 수동 순서 업데이트 (manualOrder 저장)
  - 무효 영역에 드롭: 원위치로 애니메이션 복귀
```

### 12.2 열별 드롭 가능 여부

```typescript
// 드롭 가능 여부 검증 (클라이언트 사이드)
function canDrop(
  task: Task,
  targetColumn: TaskStatus,
  allTasks: Task[]
): { allowed: boolean; warning?: string } {

  const validTransitions: Record<TaskStatus, TaskStatus[]> = {
    backlog:     ['in_progress', 'blocked'],
    in_progress: ['review', 'blocked'],
    review:      ['in_progress', 'done', 'blocked'],
    done:        ['backlog'],
    blocked:     ['backlog', 'in_progress', 'review'],
  }

  if (!validTransitions[task.status].includes(targetColumn)) {
    return { allowed: false }
  }

  // backlog → in_progress: 선행 태스크 미완료 경고
  if (targetColumn === 'in_progress') {
    const incompleteDeps = task.dependencies.filter(
      (depId) => allTasks.find((t) => t.id === depId)?.status !== 'done'
    )
    if (incompleteDeps.length > 0) {
      return { allowed: true, warning: '선행 태스크가 완료되지 않았습니다.' }
    }
  }

  // → blocked: blockedReason 모달 필요
  if (targetColumn === 'blocked') {
    return { allowed: true, warning: 'NEEDS_BLOCKED_REASON' }
  }

  return { allowed: true }
}
```

### 12.3 드롭 후 처리 흐름

```
드래그앤드롭 완료
  ↓
canDrop() 검증
  ↓
  ├── allowed: false → 원위치 복귀 + 흔들림 애니메이션
  │
  ├── warning: 'NEEDS_BLOCKED_REASON' → blockedReason 입력 모달 표시
  │     ├── 입력 완료 → 낙관적 업데이트 + 서버 요청
  │     └── 취소 → 원위치 복귀
  │
  ├── warning: '선행 태스크...' → 경고 토스트 표시 후 낙관적 업데이트 허용
  │
  └── allowed: true → 낙관적 업데이트 + 서버 PATCH 요청
        ├── 서버 응답 성공 → Firestore onSnapshot으로 동기화
        └── 서버 응답 실패 → 오류 토스트 + 원위치 복귀
```

### 12.4 모바일 터치 지원

```
터치 시작: 500ms 롱프레스로 드래그 모드 진입
  - 진동 피드백 (navigator.vibrate(50))
  - 카드 강조 표시

터치 이동: 손가락 이동 = 카드 이동

터치 끝: 드롭 영역 판정 (열 상단 절반 = 위쪽 삽입, 하단 절반 = 아래쪽 삽입)

모바일 보드 레이아웃:
  - 기본: 단일 열 표시 (PLAN.md 4.3 칸반 보드 참조)
  - 열 전환: 상단 탭 또는 가로 스와이프
  - 드래그앤드롭은 같은 열 내 순서 변경만 모바일에서 지원
  - 열 이동은 카드 상세 패널에서 상태 드롭다운으로 처리
```

## 부록 C. Firestore 인덱스

태스크 쿼리 성능을 위한 복합 인덱스:

```json
{
  "indexes": [
    {
      "collectionGroup": "tasks",
      "queryScope": "COLLECTION",
      "fields": [
        { "fieldPath": "status", "order": "ASCENDING" },
        { "fieldPath": "priority", "order": "ASCENDING" },
        { "fieldPath": "createdAt", "order": "ASCENDING" }
      ]
    },
    {
      "collectionGroup": "tasks",
      "queryScope": "COLLECTION",
      "fields": [
        { "fieldPath": "assignedSetId", "order": "ASCENDING" },
        { "fieldPath": "status", "order": "ASCENDING" }
      ]
    }
  ]
}
```

> **참고**: `tasks`는 `projects/{projectId}/tasks` 서브컬렉션이므로 인덱스에 `projectId` 필드를 포함할 필요가 없다. 쿼리는 이미 해당 프로젝트 컬렉션 경로로 스코프가 제한된다.
