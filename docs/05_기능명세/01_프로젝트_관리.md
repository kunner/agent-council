## 1. 프로젝트 개념 정의

### 1.1 프로젝트란

**프로젝트(Project)** 는 Agent Council 시스템에서 모든 작업의 최상위 단위다.
하나의 프로젝트는 반드시 하나의 Git 리포지토리에 대응하며, 그 안에서 리더들의 의사결정과 팀원들의 실제 작업이 이루어진다.

> 프로젝트 없이 Council은 "말만 하는 회의"에 불과하다.
> 프로젝트가 존재해야 비로소 코드와 결과물이 만들어진다.

### 1.2 프로젝트 구성 요소

프로젝트는 아래 5개 구성 요소로 이루어진다.

```
Project
├── Git Repository          ← 실제 소스코드가 저장되는 곳
├── Council Room            ← 리더들과 PM이 대화·의사결정하는 채팅 공간
├── Workspaces              ← Set별 독립 작업 공간 (Git worktree 기반)
│   ├── Set A → worktree (set-a/architecture)
│   ├── Set B → worktree (set-b/backend)
│   └── Set C → worktree (set-c/frontend)
├── Board                   ← 태스크 칸반 보드
└── Artifacts               ← PR, 빌드 결과물, 문서, 분석 리포트
```

#### Git Repository

- 프로젝트의 **실질적인 작업 대상**이다.
- 신규 프로젝트: 서버 로컬에 빈 Git repo를 초기화한다.
- 기존/분석 프로젝트: 원격 GitHub URL을 서버에 클론한다.
- 기본 브랜치(`main`)는 읽기 전용 참조로 취급하며, 리더 합의 후에만 머지된다.

#### Council Room

- 리더들과 PM이 참여하는 **공유 대화 공간**이다.
- 메신저 채팅방처럼 동작한다: 실시간 메시지, 순서 보장.
- 모든 메시지는 Firestore에 영속 저장된다.
- 하나의 프로젝트에는 기본적으로 하나의 메인 Council Room이 자동 생성된다.
- 필요에 따라 "설계 논의", "버그 트리아지" 등 목적별 추가 Room을 생성할 수 있다.

#### Workspaces

- 각 Agent Set에 할당되는 **격리된 코드 작업 환경**이다.
- `git worktree` 기반으로 구성되어, 같은 repo지만 독립 브랜치에서 동시에 작업한다.
- Set 내부의 Claude Code 세션은 자신의 worktree 경로에서만 작업한다.
- 경로 구조:

```
/workspace/{projectId}/
├── main/          ← 메인 브랜치 (읽기 전용 참조)
├── set-a/         ← git worktree: set-a/architecture
├── set-b/         ← git worktree: set-b/backend
└── set-c/         ← git worktree: set-c/frontend
```

#### Board

- 프로젝트의 작업 항목을 관리하는 **칸반 보드**다.
- Council 대화에서 합의된 내용이 태스크로 변환되어 등록된다.
- 컬럼: `Backlog → In Progress → Review → Done`
- 각 태스크는 담당 Set, 우선순위, 선행 의존성, 관련 브랜치/PR 정보를 포함한다.

#### Artifacts

- 프로젝트 수행 과정에서 생성된 **산출물 저장소**다.
- PR 링크, 빌드 결과, 분석 리포트, 주요 문서 등이 포함된다.
- 분석 프로젝트의 경우 결과 리포트가 주요 산출물이 된다.

### 1.3 Firestore 데이터 구조 (프로젝트 루트)

```typescript
interface Project {
  id: string
  name: string                    // "사내 메신저 시스템"
  description: string
  ownerId: string                 // Firebase Auth UID
  type: 'new' | 'existing' | 'analysis'
  status: 'planning' | 'in_progress' | 'review' | 'completed' | 'archived'
  techStack?: string[]            // ["React", "Spring Boot", "PostgreSQL"]
  createdAt: Timestamp
  updatedAt: Timestamp
}

// Firestore 컬렉션 경로
projects/{projectId}
projects/{projectId}/git/config
projects/{projectId}/rooms/{roomId}
projects/{projectId}/rooms/{roomId}/messages/{messageId}
projects/{projectId}/sets/{setId}
projects/{projectId}/sets/{setId}/logs/{logId}
projects/{projectId}/tasks/{taskId}
projects/{projectId}/pullRequests/{prId}
projects/{projectId}/snapshots/{snapshotId}
```

## 2. 프로젝트 유형별 초기화 플로우

프로젝트는 생성 목적에 따라 세 가지 유형으로 구분된다.

| 유형 | 코드 | 설명 | 코드 수정 여부 |
|---|---|---|---|
| 신규 프로젝트 | `new` | 빈 repo에서 새 프로젝트 시작 | O |
| 기존 프로젝트 | `existing` | GitHub 기존 repo에 기능 추가 | O |
| 분석 프로젝트 | `analysis` | 읽기 전용 분석만 수행 | X |

### 2.1 신규 프로젝트 (type: 'new')

#### 사용 시나리오

```
PM: "사내 메신저 시스템을 React + Spring Boot로 새로 만들고 싶어"
→ 빈 Git repo 초기화
→ 기본 디렉토리 구조 생성
→ CLAUDE.md 자동 생성
→ Council Room 생성 및 리더 입장
```

#### 초기화 시퀀스

```
┌──────┐        ┌───────────────┐        ┌─────────────┐        ┌──────────┐
│  PM  │        │ Council Server│        │  Git (local)│        │Firestore │
└──┬───┘        └───────┬───────┘        └──────┬──────┘        └────┬─────┘
   │                    │                        │                    │
   │  POST /projects    │                        │                    │
   │  { type:'new',     │                        │                    │
   │    name, techStack}│                        │                    │
   │───────────────────>│                        │                    │
   │                    │                        │                    │
   │                    │  git init              │                    │
   │                    │  /workspace/{id}/main  │                    │
   │                    │───────────────────────>│                    │
   │                    │                        │                    │
   │                    │  기본 구조 생성          │                    │
   │                    │  (README.md, .gitignore │                    │
   │                    │   CLAUDE.md)            │                    │
   │                    │───────────────────────>│                    │
   │                    │                        │                    │
   │                    │  initial commit        │                    │
   │                    │───────────────────────>│                    │
   │                    │                        │                    │
   │                    │  프로젝트 문서 저장      │                    │
   │                    │────────────────────────────────────────────>│
   │                    │                        │                    │
   │                    │  Council Room 생성      │                    │
   │                    │────────────────────────────────────────────>│
   │                    │                        │                    │

   │<───────────────────│                        │                    │
   │                    │                        │                    │
```

#### 생성되는 기본 구조

```
/workspace/{projectId}/main/
├── .gitignore
├── README.md           ← 프로젝트명, 기술스택, 개요 자동 작성
└── CLAUDE.md           ← Council 운영 규칙 자동 생성 (아래 참조)
```

#### CLAUDE.md 자동 생성 내용

```markdown
# {프로젝트명} - Agent Council 운영 규칙

## 프로젝트 개요
- 프로젝트: {프로젝트명}
- 기술 스택: {techStack}
- 생성일: {createdAt}

## 아키텍처 원칙
- [아키텍처팀 리더가 Council 대화에서 합의된 내용을 여기에 기록]

## 코드 컨벤션
- [백엔드팀/프론트팀 리더가 합의된 컨벤션을 여기에 기록]

## 주요 결정사항
- [Council 대화에서 합의된 핵심 결정을 날짜와 함께 기록]

## Set 구성
- Set A (아키텍처): DB 스키마, API 스펙, 전체 구조 설계
- Set B (백엔드): 서버 API 구현
- Set C (프론트엔드): UI 구현
- Set D (QA): 테스트 코드, 코드 리뷰
```

#### 완료 조건

- [ ] Git repo가 `/workspace/{projectId}/main/`에 초기화됨
- [ ] 초기 커밋(README.md, .gitignore, CLAUDE.md)이 main 브랜치에 존재
- [ ] Firestore에 프로젝트 문서가 `status: 'planning'`으로 저장됨
- [ ] 메인 Council Room이 생성됨 (`purpose: '메인 회의'`)

### 2.2 기존 프로젝트 (type: 'existing')

#### 사용 시나리오

```
PM: "이 GitHub 프로젝트에 파일 전송 기능을 추가하고 싶어"
    → GitHub URL: https://github.com/org/project
→ 기존 repo 클론
→ 코드 분석 (언어, 프레임워크, 구조 파악)
→ 분석 결과를 리더들에게 컨텍스트로 제공
→ 기존 코드 구조·컨벤션을 준수하며 개발 시작
```

#### 초기화 시퀀스

```

│  PM  │        │ Council Server│        │  Git (local)│        │Firestore │
└──┬───┘        └───────┬───────┘        └──────┬──────┘        └────┬─────┘
   │                    │                        │                    │
   │  POST /projects    │                        │                    │
   │  { type:'existing',│                        │                    │
   │    repoUrl,        │                        │                    │
   │    githubToken? }  │                        │                    │
   │───────────────────>│                        │                    │
   │                    │                        │                    │
   │                    │  git clone {repoUrl}   │                    │
   │                    │  /workspace/{id}/main  │                    │
   │                    │───────────────────────>│                    │
   │                    │                        │                    │
   │                    │         클론 진행률 스트리밍 (WebSocket)      │
   │<════════════════════════════════════════════════════════════════ │
   │                    │                        │                    │
   │                    │  코드 구조 분석          │                    │
   │                    │  (언어 감지, 프레임워크, │                    │
   │                    │   디렉토리 구조 파악)    │                    │
   │                    │───────────────────────>│                    │
   │                    │                        │                    │
   │                    │  프로젝트 문서 저장      │                    │
   │                    │  + 분석 결과 저장        │                    │
   │                    │────────────────────────────────────────────>│
   │                    │                        │                    │
   │                    │  Council Room 생성      │                    │
   │                    │────────────────────────────────────────────>│
   │                    │                        │                    │
   │  projectId 반환     │                        │                    │
   │<───────────────────│                        │                    │
   │                    │                        │                    │
```

#### 코드 분석 단계

클론 완료 후 Council Server가 자동으로 코드베이스를 분석하여 리더 초기 컨텍스트를 구성한다.

```typescript
interface CodebaseAnalysis {
  detectedLanguages: string[]        // ["TypeScript", "Java"]
  detectedFrameworks: string[]       // ["React", "Spring Boot"]
  directoryStructure: string         // 상위 2단계 트리 텍스트
  packageFiles: string[]             // package.json, pom.xml 등
  estimatedSize: {
    files: number
    linesOfCode: number
  }
  existingClaludeMd?: string         // 기존 CLAUDE.md 내용 (있을 경우)
  defaultBranch: string              // "main" | "master" 등
}
```

분석 결과는 리더 초기 시스템 프롬프트에 자동으로 주입되어, 리더들이 기존 코드베이스를 이해한 상태로 대화를 시작한다.

#### 완료 조건

- [ ] Git repo가 `/workspace/{projectId}/main/`에 클론됨
- [ ] 코드베이스 분석 결과가 Firestore에 저장됨
- [ ] 기존 CLAUDE.md가 있으면 읽어서 리더 컨텍스트에 포함됨
- [ ] Firestore에 프로젝트 문서가 `status: 'planning'`으로 저장됨
- [ ] 메인 Council Room이 생성됨

### 2.3 분석 프로젝트 (type: 'analysis')

#### 사용 시나리오

```
PM: "이 코드베이스의 보안 취약점을 분석해줘"
    → GitHub URL: https://github.com/org/legacy-system
→ 기존 repo 클론 (읽기 전용 모드)
→ 분석 Set 자동 구성 (보안 분석, 코드 품질, 아키텍처 리뷰 등)
→ 코드 수정 없이 리포트 산출
```

#### 초기화 시퀀스

기존 프로젝트와 동일한 클론 과정을 거치되, 추가적으로 **읽기 전용 보호** 와 **분석 Set 자동 구성** 이 적용된다.

```
클론 완료
     │
     ▼
읽기 전용 잠금 설정
     │  - worktree에 .git/hooks/pre-commit 훅 설치
     │  - 훅: 항상 "분석 프로젝트는 코드 수정 불가" 에러 반환
     │  - Council Server에서 Git write 작업 API 비활성화
     ▼
분석 Set 자동 구성
     │  PM이 목표를 입력하면 Council Server가 Set 구성 제안
     │
     ├── "보안 취약점 분석" 입력 시:
     │       Set A (보안 분석): OWASP, 인증/인가, SQL 인젝션 등
     │       Set B (의존성 분석): 취약한 패키지, 라이선스 검토
     │       Set C (결과 통합): 분석 종합 + 최종 리포트 작성
     │
     └── "코드 품질 분석" 입력 시:
             Set A (아키텍처 리뷰): 구조, 결합도, 응집도
             Set B (코드 스멜): 중복, 복잡도, 네이밍
             Set C (결과 통합): 개선 권고안 작성
```

#### 읽기 전용 보호 규칙

```typescript
// Council Server: 분석 프로젝트에서 차단되는 작업
const BLOCKED_FOR_ANALYSIS = [
  'git.commit',
  'git.push',
  'git.createBranch',
  'git.createWorktree',
  'git.merge',
  'task.createPR',
]

// 분석 프로젝트 전용 허용 작업
const ALLOWED_FOR_ANALYSIS = [
  'git.status',
  'git.log',
  'git.diff',
  'git.show',
  'artifacts.createReport',       // 리포트 생성 (별도 경로)
  'artifacts.exportMarkdown',
]
```

#### 분석 산출물 경로

분석 결과물은 원본 repo를 변경하지 않고 별도 경로에 저장된다.

```
/workspace/{projectId}/analysis/
├── security-report.md
├── code-quality-report.md
├── architecture-diagram.md
└── recommendations.md
```

Firebase Storage에도 백업 저장되어 PM이 언제든 다운로드할 수 있다.

#### 완료 조건

- [ ] Git repo가 `/workspace/{projectId}/main/`에 읽기 전용으로 클론됨
- [ ] pre-commit 훅으로 코드 수정 차단이 설정됨
- [ ] PM의 분석 목표 입력 후 분석 Set 구성이 제안됨
- [ ] 분석 산출물 저장 경로(`/analysis/`)가 생성됨

## 3. 프로젝트 생명주기 상태 전이

### 3.1 상태 정의

| 상태 | 코드 | 설명 | 진입 조건 |
|---|---|---|---|
| 기획 중 | `planning` | 프로젝트 생성 직후, 아키텍처 설계 단계 | 프로젝트 생성 시 자동 |
| 진행 중 | `in_progress` | 실제 개발/분석 작업이 시작된 상태 | 첫 번째 태스크가 `in_progress`로 전환될 때 |
| 일시정지 | `paused` | 모든 Set 세션이 중단된 일시정지 상태 | PM이 프로젝트 일시정지 액션 실행 |
| 검토 중 | `review` | 주요 작업 완료, 통합 검증 단계 | PM 또는 리더가 명시적으로 전환 |
| 완료 | `completed` | PM이 완료 처리한 상태 | PM이 완료 버튼 클릭 |
| 아카이브 | `archived` | 정리 완료, 히스토리 보존 상태 | 완료 후 아카이브 액션 |

### 3.2 상태 전이 다이어그램

```
                    ┌────────────────────────┐
                    │      [초기 생성]         │
                    └────────────┬───────────┘
                                 │
                                 ▼
                    ┌─────────────────────────┐
```

### 3.3 상태 전이 규칙 상세

#### planning → in_progress

- **자동 전환**: 태스크 중 하나라도 `in_progress`로 변경되는 순간 자동 전환
- **수동 전환**: PM이 "개발 시작" 버튼 클릭
- **사전 조건**:
  - Council Room이 생성되어 있을 것
  - 최소 1개의 Agent Set이 구성되어 있을 것
  - Git worktree가 최소 1개 이상 생성되어 있을 것

#### in_progress → review

- **수동 전환만 허용**: PM 또는 아키텍처팀 리더가 명시적으로 요청
- **권장 조건** (강제 아님):
  - 백로그의 `critical`/`high` 우선순위 태스크가 모두 `done`일 것
  - 열린 PR이 없거나 모두 머지 완료 상태일 것
- **전환 시 자동 처리**:
  - QA Set이 있는 경우, "통합 테스트 시작" 태스크 자동 생성
  - Council Room에 시스템 메시지로 알림

#### review → in_progress (역방향)

- **조건**: 버그 발견 또는 추가 작업이 필요한 경우
- **실행자**: PM만 가능
- **처리**: 버그/추가 작업 태스크를 Board에 등록 후 자동 전환

#### review → completed

- **수동 전환**: PM만 가능
- **사전 조건**:
  - 모든 태스크가 `done` 또는 `blocked` 상태일 것 (blocked 태스크는 PM이 명시적 확인)
  - 열린 PR이 0개일 것
- **전환 시 자동 처리**:
  - 결과 요약 문서 자동 생성 (섹션 6 참조)
  - 완료 시점 스냅샷 생성

#### completed → archived

- **수동 전환**: PM만 가능
- **처리**: 섹션 6 (아카이브) 절차 실행

#### 상태 전이 시 Room 동기화 규칙

프로젝트 상태가 전이될 때 소속 Council Room의 상태도 함께 동기화된다.

| 프로젝트 상태 전이 | Room 상태 변화 |
|---|---|
| → `completed` | 모든 Room이 `completed`로 전환 |
| → `paused` | 모든 Room이 `paused`로 전환 (리더 응답 중단) |
| `paused` → `in_progress` (재개) | 모든 Room이 `active`로 복원 |

### 3.4 상태별 Council Room 동작 변화

| 상태 | 메시지 전송 | 태스크 생성 | Set 작업 | Git 작업 |
|---|---|---|---|---|
| `planning` | O | O | O | O |
| `in_progress` | O | O | O | O |
| `review` | O | 제한 (버그만) | O (버그 수정) | O |
| `completed` | 읽기 전용 | X | X | X |
| `archived` | 읽기 전용 | X | X | X |

### 4.2 일반 설정

| 항목 | 타입 | 설명 | 수정 가능 |
|---|---|---|---|
| 프로젝트 이름 | `string` | 표시 이름 (최대 50자) | O |

| 프로젝트 유형 | `'new' \| 'existing' \| 'analysis'` | 생성 시 지정, 변경 불가 | X |

### 4.3 기술 스택 설정

PM이 프로젝트에서 사용하는 기술을 등록한다. 이 정보는 리더들의 시스템 프롬프트에 포함되어 기술에 맞는 조언과 코드 생성을 돕는다.

```typescript
interface TechStackConfig {
  languages: string[]        // ["TypeScript", "Java"]
  frameworks: string[]       // ["React", "Spring Boot"]
  databases: string[]        // ["PostgreSQL", "Redis"]
  infrastructure: string[]   // ["Docker", "Nginx"]
  testing: string[]          // ["Jest", "JUnit"]
  custom: string[]           // PM이 직접 추가한 기타 항목
}
```

**미리 정의된 프리셋** (선택하면 관련 스택이 자동 채워짐):

| 프리셋 | 스택 구성 |
|---|---|
| React + Spring Boot | TypeScript, React, Java, Spring Boot, PostgreSQL |
| Next.js Full Stack | TypeScript, Next.js, PostgreSQL, Prisma |
| Python API | Python, FastAPI, PostgreSQL, Redis |
| Vue + Django | TypeScript, Vue 3, Python, Django, PostgreSQL |

### 4.4 Set 구성 설정

Council 운영을 위한 Agent Set을 관리한다.

```typescript
interface SetConfig {
  id: string
  name: string               // "아키텍처팀", "백엔드팀"
  role: string               // 리더 역할 프롬프트 (상세 지시사항)
  color: string              // UI 색상 (8가지 중 선택)
  teammates: number          // 팀원 수 (1~5)
  defaultBranch: string      // 자동 생성될 브랜치명 패턴
}
```

**Set 관리 규칙:**
- 프로젝트 상태가 `planning` 또는 `in_progress`일 때만 추가/수정/삭제 가능
- `review` 이후에는 Set 구성 변경 불가
- Set 삭제 시 해당 Set의 worktree와 미완료 태스크 처리 방법을 PM에게 확인

### 4.5 Git 연동 설정

```typescript
interface GitConfig {
```

| 권한 | 용도 |

## 7. 다중 프로젝트 관리

### 7.1 사용자별 격리

모든 프로젝트는 생성한 사용자(`ownerId`)에 완전히 귀속된다.

```javascript
// Firestore Security Rules: 내 프로젝트만 접근 가능
match /projects/{projectId} {
  allow read, write: if request.auth != null
    && resource.data.ownerId == request.auth.uid;
}
```

- 다른 사용자의 프로젝트는 조회·접근 불가 (공유 기능 미지원, Phase 1)
- 사용자 삭제 시 해당 사용자의 모든 프로젝트도 연쇄 삭제 (Firebase Function으로 구현)

### 7.2 동시 실행 제한

서버 리소스(Oracle Ampere 24GB RAM)를 보호하기 위해 동시 활성 프로젝트 수를 제한한다.

#### 제한 기준

| 제한 항목 | 기본값 | 설명 |
|---|---|---|
| 사용자당 전체 프로젝트 수 | 20개 | 아카이브 포함 |
| 사용자당 활성 프로젝트 수 | 3개 | `planning` + `in_progress` + `review` 합산 |
| 프로젝트당 동시 활성 Set 수 | 8개 | Claude Code 세션 수 |
| 전체 서버 동시 Set 수 | 20개 | 전체 사용자 합산 |

#### 제한 초과 시 처리

```
활성 프로젝트 3개 초과 시도:
→ "현재 활성 프로젝트가 3개입니다.
   새 프로젝트를 시작하려면 기존 프로젝트를 일시정지하거나
   완료 처리해 주세요."
→ [일시정지할 프로젝트 선택] 버튼 제공

전체 서버 동시 Set 20개 초과 시:
→ 큐에 대기 (최대 5분)
→ 5분 초과 시 에러 반환: "서버가 혼잡합니다. 잠시 후 시도해 주세요."
```

### 7.3 프로젝트 목록 정렬 및 필터

```typescript
interface ProjectListQuery {
  filter: {
    status?: ProjectStatus | ProjectStatus[]
    type?: ProjectType
    search?: string       // 프로젝트명 검색
  }
  sort: {
    by: 'updatedAt' | 'createdAt' | 'name' | 'progress'
    order: 'desc' | 'asc'
  }
  pagination: {
    limit: number         // 기본 20
    startAfter?: string   // 커서 기반 페이지네이션
  }
}
```

기본 정렬: `updatedAt desc` (최근 활동 순)

### 7.4 프로젝트 간 리소스 공유 금지

- Set(Claude Code 세션)은 특정 프로젝트에 귀속되며 다른 프로젝트에서 재사용 불가
- worktree 경로는 `projectId`를 포함하여 격리 보장
- 한 프로젝트의 Claude Code 세션이 다른 프로젝트의 파일을 읽거나 쓰지 못하도록 작업 디렉토리를 worktree 내로 제한

## 8. 에러 케이스 처리

### 8.1 Git 클론 실패

#### 원인 분류 및 처리

| 원인 | 에러 코드 | 사용자 메시지 | 자동 조치 |
|---|---|---|---|
| 잘못된 URL 형식 | `GIT_INVALID_URL` | "올바른 GitHub 리포지토리 URL을 입력해 주세요." | 없음 |
| 리포지토리 없음 (404) | `GIT_REPO_NOT_FOUND` | "리포지토리를 찾을 수 없습니다. URL을 확인해 주세요." | 없음 |
| 인증 실패 (비공개 repo) | `GIT_AUTH_FAILED` | "비공개 리포지토리입니다. GitHub Token을 입력해 주세요." | Token 입력 폼 표시 |
| 네트워크 타임아웃 | `GIT_TIMEOUT` | "네트워크 오류가 발생했습니다. 잠시 후 다시 시도해 주세요." | 3회 자동 재시도 (10초 간격) |
| 디스크 공간 부족 | `GIT_DISK_FULL` | 섹션 8.3 참조 | 자동 정리 시도 |
| 클론 도중 중단 | `GIT_CLONE_INTERRUPTED` | "클론이 중단되었습니다. 재시도하겠습니다." | 부분 클론 삭제 후 재시도 |

#### 클론 재시도 시퀀스

```
클론 실패 감지
     │
     ├── 재시도 가능한 에러? (네트워크, 타임아웃)
     │         │
     │         ├── YES: 자동 재시도 (최대 3회, 10초 간격)
     │         │           │
     │         │           ├── 3회 내 성공: 정상 진행
     │         │           └── 3회 실패: 사용자에게 수동 재시도 버튼 제공
     │         │
     │         └── NO: 즉시 사용자에게 에러 메시지 + 해결 방법 안내
     │
     └── 부분 클론 파일 정리 (항상)
             /workspace/{projectId}/ 하위 파일 삭제
```

### 8.2 권한 없음

#### 발생 지점 및 처리

| 상황 | 에러 | 처리 |
|---|---|---|
| 다른 사용자 프로젝트 URL 직접 접근 | `FORBIDDEN` | 404 페이지로 리다이렉트 (프로젝트 존재 여부 노출 방지) |
| 로그인 없이 프로젝트 접근 | `UNAUTHENTICATED` | 로그인 페이지로 리다이렉트, 원래 URL 복귀 처리 |
| GitHub Token 없이 비공개 repo 클론 | `GIT_AUTH_FAILED` | Token 입력 폼 표시 |
| GitHub Token 권한 부족 (PR 생성 실패) | `GITHUB_INSUFFICIENT_SCOPE` | "GitHub Token에 `repo` 권한이 필요합니다." + 재발급 안내 링크 |
| 완료/아카이브 프로젝트 수정 시도 | `PROJECT_READONLY` | "이 프로젝트는 {status} 상태로 수정할 수 없습니다." |

#### Firestore Rules 에러 처리

```typescript
// 클라이언트에서 Firestore Permission Denied 처리
try {
  await setDoc(doc(db, `projects/${projectId}`), data)
} catch (error) {
  if (error.code === 'permission-denied') {
    // 서버 측 검증과 별개로 클라이언트 재확인
    await refreshAuthToken()
    // 토큰 갱신 후 재시도, 그래도 실패 시 로그아웃 처리
  }
}
```

### 8.4 세션 관련 에러

| 에러 | 설명 | 처리 |
|---|---|---|
| Claude Code 세션 시작 실패 | API 키 오류, 네트워크 등 | 에러 원인 표시 + 재시도 버튼. API 키 오류 시 설정 페이지로 안내 |
| 세션 응답 없음 (타임아웃) | Claude Code 응답이 120초 초과 | "응답 대기 중..." → 180초 초과 시 자동 세션 재시작 |
| worktree 생성 실패 | Git 상태 이상, 브랜치 충돌 | 에러 로그 표시 + Council Room에 시스템 메시지 알림 |
| PR 생성 실패 (GitHub API) | Token 만료, Rate Limit | Token 만료: 재입력 안내. Rate Limit: 1시간 후 재시도 안내 |

### 8.5 에러 로그 및 추적

모든 에러는 아래 경로에 저장되어 추후 디버깅에 활용된다.

```typescript
// Firestore: projects/{projectId}/errors/{errorId}
interface ProjectError {
  code: string               // "GIT_CLONE_INTERRUPTED" 등
  message: string            // 내부 에러 메시지
  context: Record<string, unknown>  // 발생 컨텍스트
  resolvedAt?: Timestamp     // 해결된 경우 시간 기록
  timestamp: Timestamp
}
```

Council Room에는 사용자 친화적 메시지만 표시하고, 내부 에러 상세는 설정 > 에러 로그에서 확인 가능하도록 한다.
