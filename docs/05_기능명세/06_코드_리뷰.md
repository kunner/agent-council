## 1. 개요

Agent Council의 코드 리뷰 기능은 **AI 리더들이 Council Room에서 직접 합의하고 수행하는 리뷰 프로세스**다. 전통적인 GitHub PR 리뷰와 달리, 리더들이 실시간으로 대화하며 리뷰 전략을 논의하고, 리뷰 결과를 Council Room과 GitHub PR 양쪽에 동시에 기록한다.

### 핵심 특징

- Council Room 대화가 코드 리뷰의 중심 — GitHub PR은 결과의 반영
- 리뷰어는 특정 Set(QA, 아키텍처 등)이 담당하며, Set 내부 팀원이 실제 코드를 분석
- 리뷰 코멘트는 Council 메시지와 GitHub PR 코멘트에 이중으로 기록
- 자동화 가능한 항목(코드 스타일, 보안 등)은 자동으로 선별 처리

## 2. 코드 리뷰 개념

### 2.1 Council 기반 AI 리더 간 코드 리뷰

Agent Council의 코드 리뷰는 **Set 리더들이 Council Room에서 합의하는 방식**으로 운영된다. 단순히 도구가 자동으로 코드를 검사하는 것이 아니라, 각 Set 리더가 자신의 전문성(아키텍처, QA, 보안 등)으로 서로의 코드를 검토하는 **동료 리뷰(peer review)** 구조다.

```
┌─────────────────────────────────────────────────────┐
│                    Council Room                      │
│                                                      │
│  🟢 백엔드팀 (작성자): "API 구현 완료, PR #3 올립니다" │
│                                                      │
│  🟡 QA팀 (리뷰어):    "리뷰 시작합니다. 테스트 커버리지│
│                        확인 중..."                    │
│                                                      │
│  🎯 아키텍처팀 (리뷰어): "API 스펙 일치 여부 확인 중" │
│                                                      │
│  👤 PM: "리뷰 완료되면 알려주세요"                   │
└─────────────────────────────────────────────────────┘
         │                          │
         ▼                          ▼
  Set D (QA) 내부           Set A (아키텍처) 내부
  팀원이 실제 코드           팀원이 API 스펙과
  분석 + 테스트 실행         코드 비교 분석
         │                          │
         └──────────┬───────────────┘
                    ▼
           리뷰 결과 → Council Room 보고
           리뷰 결과 → GitHub PR 코멘트 동시 기록
```

### 2.2 리뷰어 Set의 역할

각 Set은 자신의 역할에 따라 리뷰의 초점이 다르다.

| Set 역할 | 리뷰 초점 |
|---|---|
| QA팀 | 테스트 커버리지, 엣지 케이스, 회귀 테스트 가능성 |
| 아키텍처팀 | API 스펙 준수, 레이어 구조, 의존성 방향 |
| 보안팀 | 인증/인가, 입력 검증, 취약점 패턴 |
| 프론트팀 | API 계약(contract) 준수, 응답 형식 |
| DevOps팀 | 배포 영향도, 환경 변수, 리소스 사용 |

### 2.3 리뷰 단위

- **PR 단위 리뷰**: 하나의 PR 전체를 대상으로 리뷰 진행 (기본)
- **파일 단위 코멘트**: 특정 파일/라인에 대한 구체적 코멘트
- **설계 리뷰**: 코드 구현 이전, 설계 문서/API 스펙에 대한 선제적 리뷰 (선택적)

## 3. 리뷰 트리거 방식

코드 리뷰는 세 가지 방식으로 시작될 수 있다.

### 3.1 PM이 [코드 리뷰 요청] 버튼 클릭

Council Room에 PR 생성 알림이 표시될 때, 시스템 메시지에 인라인 버튼이 포함된다.

```
┌─ 시스템 메시지 ──────────────────────────────────┐
│  ⚙️ 시스템                              14:35    │
│  PR #3 "백엔드 채팅 API" 가 생성되었습니다        │
│  set-b/backend → main | +423 -12 | 파일 8개      │
│                                                   │
│  [PR 보기]  [코드 리뷰 요청]                      │
└───────────────────────────────────────────────────┘
```

PM이 [코드 리뷰 요청] 버튼을 클릭하면:
1. Council Server가 리뷰 요청 이벤트를 생성
2. 기본 리뷰어 Set 조합이 자동 제안됨 (QA + 아키텍처)
3. PM이 리뷰어 Set을 확정하거나 변경 가능
4. Council Room에 리뷰 시작 안내 메시지 게시

### 3.2 PM이 채팅으로 "리뷰해줘" 지시

PM이 자연어로 리뷰를 지시하는 경우, Council Server가 의도를 감지하여 리뷰 프로세스를 시작한다.

**감지 패턴 예시:**

```
PM: "PR #3 리뷰해줘"
PM: "백엔드 코드 한번 봐줘"
PM: "QA팀, 이번 PR 검토해줄 수 있어?"
PM: "코드 리뷰 부탁드립니다"
PM: "머지 전에 리뷰 한 번 해주세요"
```

Council Server는 메시지에서 다음을 추출한다:
- 대상 PR 번호 (명시된 경우) 또는 최신 열린 PR
- 지정 리뷰어 Set (명시된 경우) 또는 기본값 적용
- 긴급도 ("빨리", "급하게" 등 키워드)

### 3.3 Set 리더가 자발적으로 리뷰 제안

Set 리더가 스스로 다른 팀의 PR을 발견하고 리뷰를 제안할 수 있다. 이는 Council Room에서 일반 메시지로 이루어진다.

**예시 대화:**

```
🟡 QA팀 리더:
  PR #3을 확인해봤는데, 몇 가지 테스트 누락이 보입니다.
  리뷰를 진행해도 될까요?

👤 PM:
  네, 진행해주세요.

🟡 QA팀 리더:
  알겠습니다. 리뷰 시작합니다.
  [시스템이 PR #3의 리뷰 상태를 requested로 변경]
```

PM 또는 작성자 Set 리더의 동의가 있으면 Council Server가 공식 리뷰 프로세스로 전환한다.

## 4. 리뷰 프로세스 상세

### 4.1 전체 시퀀스 다이어그램

```
PM / 작성자 Set          Council Server         리뷰어 Set (QA + 아키)
       │                       │                        │
       │  리뷰 요청 트리거       │                        │
       │──────────────────────>│                        │
       │                       │  리뷰어 Set 지정        │
       │                       │──────────────────────> │
       │                       │                        │ Set 내부 작업
       │                       │                        │ 팀원에게 분배
       │                       │                        │ 코드 분석 시작
       │                       │  진행 상황 업데이트     │
       │<──────────────────────│<──────────────────────│
       │                       │                        │ 분석 완료
       │                       │  리뷰 결과 보고         │
       │<──────────────────────│<──────────────────────│
       │                       │  GitHub PR에 코멘트 기록│
       │                       │──────────────────────> │ (GitHub API)
       │                       │                        │
  [수정 필요?]                  │                        │
       │  수정 지시             │                        │
       │──────────────────────>│                        │
       │                       │  원래 Set에 수정 전달   │
       │                       │──────────────────────> │ (작성자 Set)
       │                       │                        │
       │                       │  수정 완료 + 재리뷰 요청 │
       │<──────────────────────│<──────────────────────│
       │                       │                        │ 재리뷰
       │                       │  승인                  │
       │<──────────────────────│<──────────────────────│
       │  PM 최종 확인          │                        │
       │──────────────────────>│                        │
       │                       │  PR 머지 실행           │
       │<──────────────────────│                        │
```

### 4.2 단계별 상세

#### 단계 1: PR 생성 → Council에 알림

작성자 Set 리더가 작업 완료를 보고하면 Council Server가 자동으로 PR을 생성하고 Council Room에 알린다.

```
🟢 백엔드팀 리더:
  채팅 API 구현 완료했습니다.
  - POST /api/rooms/{roomId}/messages — 메시지 전송
  - GET  /api/rooms/{roomId}/messages — 메시지 목록 (cursor 페이지네이션)
  - WebSocket /ws/rooms/{roomId}      — 실시간 연결
  총 5개 엔드포인트, 테스트 3개 작성했습니다.

⚙️ 시스템:
  PR #3 "feat: 채팅 API 구현" 이 생성되었습니다
  set-b/backend → main | +423 -12 | 파일 8개
  [PR 보기]  [코드 리뷰 요청]
```

Council Server가 수행하는 작업:
- GitHub API로 PR 생성 (sourceBranch → targetBranch)
- Firestore `pullRequests` 컬렉션에 PR 문서 생성 (status: `open`)
- Council Room에 시스템 메시지 게시
- 연결된 태스크 상태를 `in_progress` → `review`로 변경

#### 단계 2: 리뷰어 Set 지정

리뷰 요청이 접수되면 Council Server가 리뷰어 Set 조합을 결정한다.

**자동 지정 로직:**

```
PR의 변경 파일 유형을 분석:
  - .java / .ts / .py → QA Set 필수 포함
  - API 스펙 변경 감지 → 아키텍처 Set 포함
  - 인증/보안 관련 파일 → 보안 Set 포함 (있는 경우)
  - 프론트 컴포넌트 → 프론트 Set 제외 (자기 리뷰 방지)

결과: 작성자 Set과 다른 1~2개 Set을 기본 리뷰어로 제안
```

PM이 제안을 확정하거나 직접 조합을 변경할 수 있다.

```
⚙️ 시스템:
  PR #3 리뷰어 Set 제안:
  - 🟡 QA팀 (테스트 커버리지 + 기능 검증)
  - 🎯 아키텍처팀 (API 스펙 일치 확인)
  [확정]  [변경]

👤 PM:
  확정합니다.

⚙️ 시스템:
  PR #3 리뷰 시작. 리뷰어: QA팀, 아키텍처팀
  예상 소요 시간: ~10분
```

#### 단계 3: 리뷰어 Set이 팀원에게 리뷰 작업 분배

각 리뷰어 Set 내부에서 리더가 팀원들에게 작업을 분배한다. 이 과정은 Set 내부에서 이루어지며 Council Room에는 요약만 노출된다.

**Set 내부 작업 분배 예시 (QA팀):**

```
🟡 QA팀 리더 (내부):
  PR #3 리뷰 작업 분배합니다.
  - QA-1: 단위 테스트 커버리지 분석 (ChatController, ChatService)
  - QA-2: API 응답 형식 및 에러 핸들링 검증
  - QA-3: WebSocket 연결/해제 시나리오 테스트

🟡 QA팀 로그 (Council Room에 접기/펼치기로 표시):
  ├ QA-1: 테스트 파일 분석 중 🔄
  ├ QA-2: API 응답 검증 완료 ✅  (문제 없음)
  └ QA-3: WebSocket 예외 처리 누락 발견 ⚠️
```

#### 단계 4: 리뷰 결과 Council Room에 보고

각 리뷰어 Set이 분석을 완료하면 Council Room에 결과를 보고한다.

```
🟡 QA팀 리더:
  PR #3 리뷰 완료했습니다.

  ✅ 통과 항목:
  - API 응답 형식 정상 (200/201/400/404/500)
  - 메시지 전송/조회 단위 테스트 14개 통과
  - cursor 페이지네이션 엣지 케이스 처리

  ⚠️ 수정 요청 (2건):
  1. [중요] WebSocket 연결 해제 시 예외 처리 누락
     → ChatWebSocketHandler.java:45 — disconnectSession() 에서
       StaleSessionException 처리 없음
  2. [권장] 메시지 조회 API 응답에 totalCount 필드 추가 권장
     → 프론트팀 페이지네이션 UI 구현에 필요할 수 있음

🎯 아키텍처팀 리더:
  API 스펙 대조 완료.

  ✅ 통과 항목:
  - REST 엔드포인트 3개 모두 스펙 일치
  - WebSocket 경로 및 STOMP 프레임 형식 일치
  - 에러 응답 형식 공통 스펙 준수

  ℹ️ 참고 사항:
  - POST /messages 응답에 createdAt 필드가 스펙에는 ISO-8601인데
    구현에서 Unix timestamp로 반환 중 → 기능 문제는 아니나 통일 권장

👤 PM:
  수정 사항 처리 부탁드립니다.
```

#### 단계 5: 수정 요청 시 → 원래 Set이 수정 → 재리뷰

수정이 요청되면 원래 작성자 Set(백엔드팀)에게 수정 작업이 전달된다.

```
⚙️ 시스템:
  PR #3 수정 요청됨. 백엔드팀에게 수정 작업 전달합니다.

🟢 백엔드팀 리더:
  알겠습니다. 수정 시작합니다.
  1. WebSocket 예외 처리 추가 (필수)
  2. 응답 형식 ISO-8601 통일 (참고 사항 반영)

  ... (수정 작업 진행) ...

🟢 백엔드팀 리더:
  수정 완료했습니다.
  - ChatWebSocketHandler.java: StaleSessionException 처리 추가
  - MessageResponse.java: createdAt을 ISO-8601 형식으로 변경
  재리뷰 요청드립니다.

⚙️ 시스템:
  PR #3 업데이트됨 (2 커밋 추가)
  재리뷰 시작 — QA팀, 아키텍처팀
```

재리뷰는 변경된 파일만 대상으로 수행한다 (전체 재리뷰 불필요).

#### 단계 6: 승인 → PM 최종 확인 → 머지

```
🟡 QA팀 리더:
  재리뷰 완료. 수정 사항 모두 반영 확인.
  PR #3 승인합니다. ✅

🎯 아키텍처팀 리더:
  동일하게 확인 완료. 승인합니다. ✅

⚙️ 시스템:
  PR #3 모든 리뷰어 승인 완료.
  [머지하기]  [대기]

👤 PM:
  머지해주세요.

⚙️ 시스템:
  PR #3 main에 머지 완료.
  Set A, Set C worktree 자동 rebase 중...
  ✅ Set A: rebase 성공
  ✅ Set C: rebase 성공
  태스크 "채팅 API 구현" → Done
```

## 5. 리뷰 코멘트 이중 기록

모든 리뷰 코멘트는 **Council Room 메시지**와 **GitHub PR 코멘트** 양쪽에 동시에 기록된다.

### 5.1 이중 기록 구조

```
┌─────────────────────────────────────────────────────┐
│  리뷰어 Set이 Council Room에 리뷰 결과 보고            │
│  (자연어 형식, 팀원들이 읽기 쉬운 형태)                │
└─────────────────────────┬───────────────────────────┘
                          │
              Council Server가 처리
                          │
          ┌───────────────┴───────────────┐
          ▼                               ▼
┌──────────────────┐           ┌─────────────────────┐
│  Firestore       │           │  GitHub PR           │
│  messages 컬렉션  │           │  (GitHub API)        │
│                  │           │                      │
│  - 전체 리뷰 요약  │           │  - 파일별 인라인 코멘트│
│  - 메시지 형태    │           │  - PR 전체 코멘트     │
│  - 이모지/서식 유지│           │  - approve/request  │
│                  │           │    changes 상태 반영  │
└──────────────────┘           └─────────────────────┘
```

### 5.2 GitHub PR 코멘트 변환 규칙

Council Room의 리뷰 코멘트를 GitHub PR 코멘트로 변환할 때 적용되는 규칙:

| Council 코멘트 유형 | GitHub 변환 형식 |
|---|---|
| 특정 파일:라인 언급 | GitHub 인라인 코멘트 (파일+라인 지정) |
| 전체 평가 (통과/수정 요청) | PR 전체 리뷰 코멘트 |
| 승인 | GitHub PR approve |
| 수정 요청 항목 포함 | GitHub `request changes` |
| 참고/권장 사항만 | GitHub approve with comment |

### 5.3 GitHub PR 코멘트 형식


## 6. 자동 리뷰 항목

리뷰어 Set이 코드를 분석할 때, 특정 항목은 자동화된 도구로 선별 처리하고 결과를 리뷰 보고에 포함한다.

### 6.1 자동 처리 항목 목록

#### 코드 스타일 검사

```
검사 도구 (언어별 자동 선택):
  Java/Kotlin  → Checkstyle, Google Java Format
  TypeScript   → ESLint, Prettier
  Python       → Flake8, Black
  Go           → gofmt, golint

검사 항목:
  - 들여쓰기 및 공백 규칙
  - 명명 규칙 (camelCase, PascalCase 등)
  - 최대 줄 길이
  - 미사용 임포트/변수
  - 주석 형식
```

#### 테스트 커버리지 분석

```
측정 기준:
  - 라인 커버리지 (목표: 70% 이상)
  - 브랜치 커버리지 (목표: 60% 이상)
  - 변경된 코드의 커버리지 (신규 코드 기준: 80% 이상 권장)

자동 판단:
  - 커버리지 기준 미달 시 ⚠️ 경고
  - 테스트 없는 public 메서드 목록 추출
  - 기존 대비 커버리지 하락 감지 시 알림
```

#### 보안 취약점 스캔

```
검사 항목:
  - SQL Injection 패턴 (문자열 쿼리 조합 감지)
  - XSS 취약점 (미검증 사용자 입력 출력)
  - 하드코딩된 비밀값 (API 키, 패스워드, 토큰)
  - 취약한 암호화 알고리즘 사용 (MD5, SHA1)
  - 인증 우회 패턴 (주석 처리된 인증 코드)
  - 의존성 취약점 (CVE 데이터베이스 대조)

심각도 분류:
  🔴 Critical — 즉시 수정 필요, 머지 블록
  🟠 High     — 수정 요청 (머지 전)
  🟡 Medium   — 권장 수정
  🔵 Low      — 참고 사항
```

### 6.2 자동 리뷰 결과 보고 형식

```
⚙️ 자동 리뷰 결과 (PR #3)

📏 코드 스타일: ✅ 통과
   - ESLint: 0 errors, 2 warnings (비치명적)
   - Prettier: 모든 파일 형식 준수

🧪 테스트 커버리지:
   - 전체 커버리지: 74.2% (기준: 70%) ✅
   - 변경 파일 커버리지: 82.1% ✅
   - 커버리지 없는 public 메서드: WebSocketHandler.cleanup() ⚠️

🔒 보안 스캔:
   - 발견된 취약점: 1건
   🟡 Medium: application.properties에 개발용 토큰 하드코딩
              → 환경 변수로 분리 권장
```

### 6.3 자동 리뷰와 수동 리뷰의 관계

```
자동 리뷰 먼저 실행 (리뷰어 Set이 코드 분석 시작과 동시에)
        │
        ▼
자동 리뷰 결과를 리뷰어 Set에게 전달
        │
        ▼
리뷰어 Set이 자동 결과를 참고하여 수동 분석 수행
        │
        ▼
최종 리뷰 보고 = 자동 결과 요약 + 수동 분석 결과 통합
```

Critical 보안 취약점이 발견된 경우, 자동으로 머지가 블록되고 작성자 Set에게 즉시 알림이 전달된다.

## 7. 리뷰 상태 추적

### 7.1 리뷰 상태 전이도

```
      [PR 생성]
          │
          ▼
      requested ──── (리뷰어 지정 완료)
          │
          ▼
      in_review ──── (리뷰어 Set이 코드 분석 중)
          │
          ├──── (수정 사항 있음) ────> changes_requested
          │                                   │
          │                         (작성자 Set 수정 완료)
          │                                   │
          │                            in_review (재리뷰)
          │                                   │
          └──── (수정 사항 없음) ─────────────┘
                                              │
                                              ▼
                                          approved ──── (PM 확인)
                                                             │
                                                        [머지 실행]
                                                             │
                                                             ▼
                                                          merged
```

### 7.2 상태별 정의

| 상태 | 설명 | 가능한 다음 상태 |
|---|---|---|
| `requested` | 리뷰 요청됨, 리뷰어 Set 지정 대기 또는 완료 | `in_review` |
| `in_review` | 리뷰어 Set이 코드 분석 중 | `changes_requested`, `approved` |
| `changes_requested` | 수정 요청 발생, 작성자 Set 수정 대기 중 | `in_review` (재리뷰 시) |
| `approved` | 모든 리뷰어 Set 승인 완료, PM 머지 대기 | `merged` |
| `merged` | PR이 main에 머지 완료 | (최종 상태) |
| `closed` | 리뷰 없이 PR 닫힘 (취소/드롭) | (최종 상태) |

### 7.3 PR status ↔ reviewStatus 동기화 매트릭스

`status`(PR 수명주기)와 `reviewStatus`(리뷰 진행 상태)는 독립적이지만 의미상 연계된다.

| status | reviewStatus | 해석 |
|---|---|---|
| `open` | `requested` | 리뷰 요청됨, 아직 리뷰 시작 전 |
| `open` | `in_review` | 리뷰어 Set이 코드 분석 중 |
| `open` | `changes_requested` | 수정 요청 발생, 작성자 Set 수정 대기 |
| `open` | `approved` | 모든 리뷰어 승인 완료, PM 머지 대기 |
| `merged` | `approved` | 머지 완료 (reviewStatus는 반드시 `approved`여야 함) |
| `closed` | (any) | PR이 머지 없이 닫힘 |

> **규칙**: `status: merged`로 전환될 때 `reviewStatus`가 `approved`가 아니면 오류로 처리한다.

### 7.4 Firestore 상태 업데이트

```typescript
// Firestore: projects/{projectId}/pullRequests/{prId}
{
  status: 'in_review',
  reviewers: ['set-d-qa', 'set-a-architecture'],
  reviewRounds: [
    {
      round: 1,
      startedAt: Timestamp,
      completedAt: Timestamp,
      results: [
        {
          setId: 'set-d-qa',
          verdict: 'changes_requested',
          commentCount: 2,
          autoReviewPassed: true
        },
        {
          setId: 'set-a-architecture',
          verdict: 'approved',
          commentCount: 1,
          autoReviewPassed: true
        }
      ]
    }
  ]
}
```

## 8. 리뷰 메트릭

### 8.1 수집 메트릭

각 PR의 리뷰 완료 후 다음 메트릭이 Firestore에 기록된다.

| 메트릭 | 설명 | 단위 |
|---|---|---|
| `reviewDuration` | 리뷰 요청 → 최종 승인까지 소요 시간 | 분 |
| `roundCount` | 총 리뷰 라운드 수 (1회 통과 = 1, 수정 후 재리뷰 = 2+) | 횟수 |
| `commentCount` | 리뷰어가 남긴 코멘트 총 수 | 건 |
| `autoReviewPassRate` | 자동 리뷰 항목 통과율 | % |
| `fixTurnaroundTime` | 수정 요청 → 수정 완료까지 소요 시간 | 분 |
| `linesReviewed` | 리뷰 대상 코드 변경 라인 수 | 라인 |
| `reviewerSets` | 리뷰에 참여한 Set 수 | 개 |

### 8.2 메트릭 저장 구조

```typescript
// Firestore: projects/{projectId}/pullRequests/{prId}/metrics
interface ReviewMetrics {
  prId: string
  reviewDurationMinutes: number
  roundCount: number
  commentCount: number
  autoReviewPassRate: number       // 0.0 ~ 1.0
  fixTurnaroundMinutes?: number    // 수정 라운드가 있는 경우만
  linesReviewed: number
  reviewerSetCount: number
  finalVerdict: 'approved' | 'closed'
  completedAt: Timestamp
}
```

### 8.3 프로젝트 리뷰 요약

프로젝트 단위로 리뷰 메트릭 집계가 제공된다.

```
┌─ 프로젝트 리뷰 요약 ──────────────────────┐
│                                            │
│  완료된 리뷰: 8건                           │
│                                            │
│  평균 리뷰 시간: 14분                       │
│  1회 통과율: 62% (5/8)                     │
│  평균 수정 라운드: 1.4회                    │
│  평균 코멘트 수: 3.2건/PR                  │
│  자동 리뷰 통과율: 89%                     │
│                                            │
│  가장 많이 발견된 이슈:                     │
│  1위. 테스트 커버리지 부족 (4건)            │
│  2위. 예외 처리 누락 (3건)                 │
│  3위. 코드 스타일 위반 (2건)               │
└────────────────────────────────────────────┘
```

### 8.4 메트릭 활용

리뷰 메트릭은 다음 용도로 활용된다.

- **프로세스 개선**: 리뷰 시간이 길어지는 원인 분석 (라운드 수, 코멘트 유형)
- **Set 성숙도 평가**: 특정 Set의 코드 품질 추이 확인
- **리뷰어 부하 분산**: 특정 Set에 리뷰가 집중되는 경우 감지
- **세션 비용 최적화**: 리뷰에 소요된 토큰 대비 발견 이슈 수 분석

## 9. 데이터 모델

### 9.1 PR 문서 확장

기존 `pullRequests` 컬렉션 문서에 리뷰 관련 필드를 추가한다.

```typescript
// Firestore: projects/{projectId}/pullRequests/{prId}
interface PullRequest {
  // 기존 필드
  id: string
  title: string
  githubPrNumber?: number
  githubPrUrl?: string
  sourceBranch: string
  targetBranch: string
  setId: string                         // 작성자 Set
  status: 'open' | 'reviewing' | 'approved' | 'merged' | 'closed'
  relatedTaskIds: string[]
  createdAt: Timestamp
  mergedAt?: Timestamp

  // 리뷰 관련 추가 필드
  reviewStatus: ReviewStatus            // 리뷰 세부 상태
  reviewerSetIds: string[]              // 지정된 리뷰어 Set ID 목록
  reviewRounds: ReviewRound[]           // 리뷰 라운드 이력
  autoReviewResult?: AutoReviewResult   // 자동 리뷰 결과
  reviewRequestedAt?: Timestamp         // 리뷰 요청 시각
  reviewCompletedAt?: Timestamp         // 최종 승인 시각
}

type ReviewStatus =
  | 'requested'
  | 'in_review'
  | 'changes_requested'
  | 'approved'
  | 'merged'
  | 'closed'

interface ReviewRound {
  round: number
  startedAt: Timestamp
  completedAt?: Timestamp
  results: ReviewRoundResult[]
}

interface ReviewRoundResult {
  setId: string
  verdict: 'approved' | 'changes_requested' | 'commented'
  commentCount: number
  councilMessageId: string              // Council Room 메시지 참조
  githubReviewId?: number               // GitHub Review ID
  submittedAt: Timestamp
}

interface AutoReviewResult {
  style: { passed: boolean; issues: string[] }
  coverage: { percentage: number; passed: boolean; uncoveredMethods: string[] }
  security: { issues: SecurityIssue[] }
  completedAt: Timestamp
}

interface SecurityIssue {
  severity: 'critical' | 'high' | 'medium' | 'low'
  description: string
  file?: string
  line?: number
}
```
