## 1. 개요

PM은 언제든 브라우저를 닫거나 자리를 비울 수 있다. 그러나 Agent Council의 프로젝트는 그 사이에도 지속되어야 하고, PM이 돌아왔을 때 대화와 맥락이 그대로 복원되어야 한다.

이를 위해 세션 연속성을 **3개 레이어**로 보장한다.

```
Layer 1: 대화 히스토리        — Firestore에 영구 저장, 브라우저 재접속 시 자동 복원
Layer 2: 프로젝트 상태 스냅샷 — 이벤트/주기 기반으로 프로젝트 현황 요약 저장
Layer 3: Claude Code 세션     — tmux 유지(전략 A) 또는 컨텍스트 재주입(전략 B)
```

3개 레이어는 독립적으로 동작하며, 상위 레이어가 실패해도 하위 레이어가 폴백을 제공한다.

## 2. Layer 1: 대화 히스토리

### 2.1 원리

모든 Council Room 메시지는 Firestore에 즉시 저장된다. Claude Code 세션이 종료되거나 서버가 재시작되어도 대화 내용은 영구 보존된다.

```
PM 메시지 전송
      │
      ▼
Council Server
      │
      ├─→ Firestore: projects/{projectId}/rooms/{roomId}/messages/{msgId} (영구 저장)
      │
      └─→ Claude Code 세션으로 전달 (휘발성)
```

### 2.2 브라우저 재접속 시 UI 자동 복원

PM이 브라우저를 닫았다가 다시 열면, Firestore `onSnapshot` 리스너가 즉시 활성화되어 이전 메시지를 자동으로 불러온다.

```typescript
// packages/web/src/hooks/useMessages.ts

function subscribeToMessages(
  db: Firestore,
  projectId: string,
  roomId: string,
  onNewMessage: (msg: Message) => void
): Unsubscribe {
  const q = query(
    collection(db, `projects/${projectId}/rooms/${roomId}/messages`),
    orderBy('timestamp')
  )
  return onSnapshot(q, (snapshot) => {
    snapshot.docChanges().forEach((change) => {
      if (change.type === 'added') {
        onNewMessage(change.doc.data() as Message)
      }
    })
  })
}
```

- 오프라인 상태였다가 복구되어도 Firestore SDK가 자동으로 동기화한다.
- 별도 재로드나 새로고침 없이 이전 대화가 즉시 표시된다.

### 2.3 메시지 페이지네이션 (cursor 기반)

대화가 길어질 경우 전체 메시지를 한 번에 불러오는 것은 성능 부담이 크다. cursor 기반 페이지네이션으로 부분 로드한다.

```typescript
// packages/web/src/hooks/useMessagePagination.ts

const PAGE_SIZE = 50

async function loadMessagesBefore(
  db: Firestore,
  projectId: string,
  roomId: string,
  beforeTimestamp: Timestamp | null   // null이면 가장 최신부터
): Promise<{ messages: Message[]; hasMore: boolean }> {
  let q = query(
    collection(db, `projects/${projectId}/rooms/${roomId}/messages`),
    orderBy('timestamp', 'desc'),
    limit(PAGE_SIZE + 1)
  )

  if (beforeTimestamp) {
    q = query(q, startAfter(beforeTimestamp))
  }

  const snapshot = await getDocs(q)
  const docs = snapshot.docs

  return {
    messages: docs.slice(0, PAGE_SIZE).map((d) => d.data() as Message).reverse(),
    hasMore: docs.length > PAGE_SIZE,
  }
}
```

- 초기 로드: 최근 50개 메시지
- "이전 메시지 불러오기" 버튼: cursor를 이동하여 이전 50개 추가 로드
- 실시간 리스너는 최신 메시지에만 유지 (페이지네이션 범위 밖은 정적 로드)

### 2.4 Layer 1의 한계

대화 내용은 복원되지만, **Claude Code 세션 내부 컨텍스트**는 복원되지 않는다. 즉, 세션이 재시작되면 리더는 이전 대화를 모르는 "빈 상태"로 시작된다. 이를 Layer 2~3이 보완한다.

## 3. Layer 2: 프로젝트 상태 스냅샷

### 3.1 스냅샷 데이터 구조

Firestore 저장 경로: `projects/{projectId}/snapshots/{snapshotId}`

(참조: [설정 참조표 섹션 3](../00_설정_참조표.md#3-firestore-컬렉션-경로))

```typescript
interface ProjectSnapshot {
  snapshotId: string
  projectId: string
  createdAt: Timestamp
  trigger: SnapshotTrigger        // 생성 원인 (아래 참조)
  triggeredBy?: string            // 'system' 또는 userId

  // 프로젝트 요약 (세션 복원 시 리더에게 주입)
  summary: string                 // "완료된 태스크 3개, 진행 중 2개. main 5커밋."
  phase: string                   // "개발 진행 중 (완료 3개)"

  // 태스크 현황
  completedTasks: CompletedTaskRef[]
  inProgressTasks: InProgressTaskRef[]

  // Council 대화에서 추출한 주요 결정사항
  decisions: DecisionRecord[]

  // Git 상태
  gitState: {
    mainCommitCount: number
    mainLastCommitMessage: string
    openPRs: { prId: string; title: string; number?: number; sourceBranch: string }[]
    branches: { name: string; setId: string; aheadCount: number }[]
  }

  // Set별 현황
  setStates: {
    setId: string
    setName: string
    status: SetStatus
    currentTaskTitle?: string
    worktreePath: string          // /opt/agent-council/workspace/{projectId}/set-{id}
    branch: string
  }[]

  // 컨텍스트 재주입용 메시지 참조
  recentMessageIds: string[]      // 최근 30개 메시지 ID
  keyMessageIds: string[]         // 결정이 담긴 핵심 메시지 ID
}

// 설정 참조표 섹션 9 기준
type SnapshotTrigger =
  | 'pr_merged'       // PR 머지 시
  | 'task_done'       // 태스크 완료 시
  | 'manual'          // PM 수동 요청
  | 'scheduled'       // 30분 주기
  | 'session_end'     // 세션 종료 직전
  | 'pm_away'         // PM 2시간 부재 시
```

### 3.2 스냅샷 생성 트리거

#### 이벤트 기반 트리거

| 이벤트 | trigger 값 | 이유 |
|---|---|---|
| PR이 main에 머지됨 | `pr_merged` | 코드베이스 상태가 변경되는 중요 체크포인트 |
| 태스크가 `done`으로 전환 | `task_done` | 작업 단위 완료, 명확한 진행 기록 |
| PM이 "상태 저장" 요청 | `manual` | PM 명시 의사 |
| PM 부재 2시간 초과 | `pm_away` | 세션 종료 전 마지막 상태 보존 |
| 세션 종료 직전 | `session_end` | Room 일시정지 또는 종료 처리 중 |

#### 주기적 트리거

- 세션이 `active` 또는 `idle` 상태일 때 **30분마다** 자동 생성 (`scheduled`)
- 단, 직전 스냅샷으로부터 **10분 이내**이면 스킵 (중복 방지)

### 3.3 최신 스냅샷 조회

```typescript
// packages/server/src/session/SnapshotService.ts

async getLatestSnapshot(projectId: string): Promise<ProjectSnapshot | null> {
  // projects/{projectId}.latestSnapshotId 필드를 통해 O(1) 조회
  const projectDoc = await getDoc(doc(firestore, `projects/${projectId}`))
  const latestSnapshotId = projectDoc.data()?.latestSnapshotId
  if (!latestSnapshotId) return null

  const snap = await getDoc(
    doc(firestore, `projects/${projectId}/snapshots/${latestSnapshotId}`)
  )
  return snap.exists() ? (snap.data() as ProjectSnapshot) : null
}
```

스냅샷이 없으면 현재 Firestore 상태(태스크, 메시지, Git)에서 즉석 생성하여 반환한다.

### 3.4 보존 정책

- 최근 **20개** 스냅샷 유지
- 21번째 스냅샷 생성 시 가장 오래된 것 자동 삭제
- `pr_merged`, `manual` 트리거 스냅샷은 **보존 대상에서 제외** (명시적 체크포인트이므로 별도 관리)

```typescript
async pruneOldSnapshots(projectId: string): Promise<void> {
  const MAX_SNAPSHOTS = 20
  const q = query(
    collection(firestore, `projects/${projectId}/snapshots`),
    orderBy('createdAt', 'desc'),
    offset(MAX_SNAPSHOTS)          // 20번째 이후부터
  )
  const old = await getDocs(q)
  const batch = writeBatch(firestore)
  old.docs
    .filter((d) => !['pr_merged', 'manual'].includes(d.data().trigger))
    .forEach((d) => batch.delete(d.ref))
  await batch.commit()
}
```

## 4. Layer 3: Claude Code 세션 복원

### 4.1 복원 우선순위

PM이 프로젝트에 접속하면 아래 순서로 세션 복원을 시도한다.

```
PM 접속
   │
   ▼
tmux 세션 생존 확인
(tmux has-session -t council-{projectId}-{setId})
   │
   ├── 살아있음 → [전략 A] 기존 세션에 바로 연결
   │
   └── 죽어있음
          │
          ├── 스냅샷 있음 → [전략 B] 스냅샷 로드 후 새 세션 생성 + 컨텍스트 주입
          │
          └── 스냅샷 없음 → 메시지에서 즉석 스냅샷 생성 → [전략 B]
```

### 4.2 전략 A: tmux 세션 유지

PM이 없어도 Claude Code 프로세스를 서버의 tmux 세션으로 유지한다.

#### tmux 세션 명명

설정 참조표 섹션 5 기준:

```
council-{projectId}-{setId}

예시:
  council-proj_abc123-set_arch    ← 아키텍처팀
  council-proj_abc123-set_back    ← 백엔드팀
  council-proj_abc123-set_front   ← 프론트팀
```

#### 세션 생존 확인

```typescript
// packages/server/src/session/TmuxSessionManager.ts

isAlive(projectId: string, setId: string): boolean {
  const name = `council-${projectId}-${setId}`
  try {
    execSync(`tmux has-session -t ${name}`, { stdio: 'ignore' })
    return true
  } catch {
    return false
  }
}
```

#### PM 복귀 시 연결 시퀀스

```
PM Browser        Council Server        tmux Session
    │                   │                    │
    │── 접속 요청 ──────→│                    │
    │                   │── isAlive? ────────→│
    │                   │←── true ────────────│
    │                   │                    │
    │                   │── 최근 로그 조회 ──→│
    │                   │←── 출력 버퍼 ───────│
    │                   │                    │
    │←── 세션 상태 전달 ──│                    │
    │    (ACTIVE 전환)   │                    │
    │                   │                    │
    │── 메시지 전송 ─────→│                    │
    │                   │── 전달 ────────────→│
    │                   │←── 응답 스트림 ──────│
    │←── 응답 표시 ───────│                    │
```

- 장점: 컨텍스트 100% 보존, 재시작 없음
- 단점: 서버 리소스 점유 (세션당 ~1.5–2GB, 설정 참조표 섹션 8), 장시간 방치 시 낭비

### 4.3 전략 B: 세션 재생성 + 컨텍스트 주입

세션이 종료된 경우 새 tmux 세션을 만들고 스냅샷 기반 컨텍스트를 시스템 프롬프트에 주입한다.

#### 복원 프롬프트 템플릿

```

역할: {setRole}

{snapshot.summary}
현재 단계: {snapshot.phase}

=== 완료된 작업 ===
- {task.title} (완료: {task.completedAt})
...

=== 진행 중인 작업 ===
- {task.title} ({task.setName}, {task.progress})
...

=== 주요 결정사항 ===
- {decision.content}
...

=== Git 상태 ===
main 브랜치: {gitState.mainCommitCount}개 커밋
현재 브랜치: {currentBranch}
열린 PR:
- #{pr.number} {pr.title}
...

=== 최근 대화 (마지막 10개) ===
[{message.senderName}] {message.content}
...

=== 지침 ===

- 이미 완료된 작업을 다시 제안하지 마세요.
- 진행 중인 작업의 현재 상태를 기준으로 응답하세요.
- 의사결정이 필요하면 PM에게 먼저 확인 후 진행하세요.
- 작업 공간: /opt/agent-council/workspace/{projectId}/set-{setId}
```

#### 전략 B 실행 시퀀스

```
PM Browser    Council Server    Firestore    새 Claude 세션
    │               │               │               │
    │── 접속 요청 ──→│               │               │
    │               │── 스냅샷 조회 →│               │
    │               │←── 스냅샷 ─────│               │
    │               │── 최근 메시지 →│               │
    │               │←── 메시지 ─────│               │
    │               │                               │
    │               │── 복원 프롬프트 생성            │
    │               │── tmux new-session ───────────→│
    │               │               │         초기화 │
    │               │←──────────────────── READY ───│
    │               │                               │
    │←── 복원 완료 ──│                               │
    │   (ACTIVE 전환)│                               │
    │               │                               │
    │── 메시지 전송 ─→│──────────────────────────────→│
    │←── 응답 ────────────────────────────────────── │
```

## 5. PM 부재 시 자동 관리 정책

PM의 마지막 활동 시각을 기준으로 세션 동작 모드가 자동 전환된다.

```
┌───────────────┬────────────────────────────────────────────────┐
│  부재 시간    │  동작                                           │
├───────────────┼────────────────────────────────────────────────┤
│  0 ~ 30분    │ IDLE 상태                                       │
│               │ - 세션 유지 (tmux 프로세스 생존)                │
│               │ - 진행 중인 작업 계속 수행                      │
│               │ - 의사결정 필요 항목은 pendingDecisions 큐에 보관│
│               │ - heartbeat 30초 간격 유지                      │
├───────────────┼────────────────────────────────────────────────┤
│  30분 ~ 2시간│ SLEEP 상태                                      │
│               │ - 세션 유지 (tmux 프로세스 생존)                │
│               │ - 신규 작업 시작 금지                           │
│               │ - 현재 진행 중인 작업만 마무리                  │
│               │ - 30분마다 scheduled 스냅샷 저장                │
│               │ - heartbeat 5분 간격으로 축소                   │
├───────────────┼────────────────────────────────────────────────┤
│  2시간 초과   │ STOPPING → STOPPED 전환                        │
│               │ - pm_away 스냅샷 저장 후 tmux 세션 종료         │
│               │ - 리소스 해제                                   │
│               │ - PM 복귀 시 전략 B로 재개                      │
└───────────────┴────────────────────────────────────────────────┘
```

### 5.1 SessionStatus 전이

설정 참조표 섹션 9 기준 `SessionStatus`:

```
starting → active ──────────────────────┐
                                         │ PM 30분 부재
                                         ▼
                                        idle
                                         │ PM 추가 30분 부재 (총 30분~)
                                         ▼
                                        sleeping
                                         │ PM 2시간 초과
                                         ▼
                               stopping → stopped
                                         │ PM 복귀 후 복원 요청
                                         ▼
                                        starting
```

### 5.2 의사결정 큐 (pendingDecisions)

PM 부재 중 리더가 의사결정이 필요한 상황에 처하면, 작업을 중단하고 큐에 등록한다.

Firestore 경로: `projects/{projectId}/pendingDecisions/{decisionId}`

```typescript
interface PendingDecision {
  id: string
  projectId: string
  setId: string
  setName: string
  question: string              // "Redis 도입 여부를 결정해야 합니다"
  context: string               // 배경 설명
  options: string[]             // ["SimpleBroker 유지", "Redis 도입"]
  impact: 'critical' | 'high' | 'medium' | 'low'
  createdAt: Timestamp
  relatedTaskId?: string
}
```

PM 복귀 시 대기 중인 항목이 있으면 Council Room에 시스템 메시지로 요약 표시된다.

```
[시스템] PM 복귀 알림: 부재 중 2개의 의사결정 대기 항목이 있습니다.
```

## 6. PM Heartbeat 메커니즘

브라우저(클라이언트)와 서버 사이의 WebSocket ping으로 PM 접속 상태를 실시간 감지한다.

### 6.1 흐름

```
브라우저              Council Server          Claude Code 세션
    │                      │                       │
    │── ping (30s) ────────→│                       │
    │   { projectId,        │── isAlive() ─────────→│
    │     userId,           │←── true/false ─────── │
    │     timestamp }       │                       │
    │                       │ PM 활동 기록 갱신      │
    │←── pong ──────────────│                       │
    │   { sessions: [...],  │                       │
    │     timestamp }       │                       │
    │                       │                       │
    │  (세션이 죽어있는 경우)                         │
    │←── session_dead ──────│                       │
    │   { deadSetIds: [...] }│                       │
    │   → 복원 절차 시작     │                       │
```

### 6.2 heartbeat 간격

| 세션 상태 | heartbeat 간격 |
|---|---|
| `active` | 30초 |
| `idle` | 30초 |
| `sleeping` | 5분 |
| `stopped` | 전송 안 함 |

### 6.3 브라우저 탭 종료 감지

`beforeunload` 이벤트로 의도적 종료를 감지하고 서버에 알린다. 이후 heartbeat가 두 번 연속 누락되면 비정상 종료로 판단한다.

```typescript
// packages/web/src/hooks/useHeartbeat.ts

useEffect(() => {
  const interval = setInterval(() => {
    ws.send(JSON.stringify({ type: 'ping', projectId, userId, timestamp: Date.now() }))
  }, HEARTBEAT_INTERVAL_MS)

  const handleUnload = () => {
    ws.send(JSON.stringify({ type: 'pm_leaving', projectId, userId }))
  }
  window.addEventListener('beforeunload', handleUnload)

  return () => {
    clearInterval(interval)
    window.removeEventListener('beforeunload', handleUnload)
  }
}, [projectId, userId])
```

## 8. 수동 세션 관리

### 8.1 수동 스냅샷 저장

PM이 언제든 현재 상태를 명시적으로 저장할 수 있다.

- **방법 1**: Council Room 입력창에 "상태 저장해줘" 또는 "체크포인트 만들어" 입력
- **방법 2**: 설정 패널 → "현재 상태 저장" 버튼 클릭

```
[시스템] 상태 스냅샷이 저장되었습니다.
         저장 시각: 2026-03-23 14:30
         태스크: 완료 3개 / 진행 2개
         Git: main 5커밋, PR 1개 열림
```

### 8.2 세션 종료 및 재개

PM이 명시적으로 세션을 종료하거나 재개할 수 있다.

#### 세션 종료

설정 패널 → "세션 종료" 버튼:

1. `session_end` 스냅샷 저장
2. 모든 tmux 세션 종료
3. Council Room에 시스템 메시지 기록
4. 프로젝트 상태 → `paused`

```
[시스템] PM이 세션을 종료했습니다.
         현재 상태가 저장되었습니다. 언제든 재개할 수 있습니다.
```

#### 세션 재개

프로젝트 목록에서 일시정지된 프로젝트의 "재개" 버튼 클릭:

1. 최신 스냅샷 로드
2. 전략 A 또는 B로 모든 세션 복원
3. 프로젝트 상태 → `in_progress`
4. Council Room에 복원 완료 메시지 기록

## 9. 에러 케이스

### 9.1 서버 크래시 시 자동 복구

```
Council Server 재시작 후 자동 복구 흐름

  서버 재시작
       │
       ▼
  active/in_progress 프로젝트 전체 조회 (Firestore)
       │
       ▼
  For each project:
    │
    ├── tmux 세션 생존 확인
    │     ├── 생존 → 전략 A: 세션 레퍼런스 재연결
    │     └── 사망 → 전략 B: 최신 스냅샷 기반 재생성
    │
    ├── heartbeat 모니터 재시작
    │
    └── Council Room에 복구 알림
         "[시스템] 서버가 재시작되었습니다. 세션을 복구합니다..."
         "[시스템] 백엔드팀 세션 복구 완료 (스냅샷: 2시간 전 기준)"
         "[시스템] 프론트팀 세션 복구 완료 (세션 유지됨)"
```

서버 크래시 시 `session_end` 스냅샷을 저장하지 못할 수 있다. 이 경우 가장 최근의 `scheduled` 또는 `task_done` 스냅샷을 사용한다.

### 9.2 Firestore 장애 시 로컬 캐시 활용

| 장애 유형 | 대응 |
|---|---|
| Firestore 일시 단절 | SDK 내장 오프라인 큐 자동 활용, 복구 시 자동 동기화 |
| Firestore 지속 장애 | 서버 메모리 내 마지막 스냅샷 임시 보관, 파일 시스템 백업 (`/opt/agent-council/logs/snapshots/`) |
| 스냅샷 조회 실패 | 메시지 컬렉션에서 즉석 요약 생성 후 전략 B 진행 |

로컬 백업 경로는 설정 참조표 섹션 2 기준: `/opt/agent-council/logs/`.

### 9.3 tmux 세션 복원 실패

```
tmux 세션 생성 실패 시:
  1. 에러 로그 기록 (/opt/agent-council/logs/)
  2. Council Room에 시스템 에러 메시지 표시
     "[시스템] 백엔드팀 세션 복원에 실패했습니다. 수동 재시작이 필요합니다."
  3. 세션 상태 → 'error'
  4. TopBar 배지: ⚠ 오류 (빨강)
  5. PM이 "재시작" 버튼으로 재시도 가능
```

동시 세션 수 상한(설정 참조표 섹션 8: 최대 8개)을 초과하는 경우, 가장 오래 유휴 상태인 세션을 먼저 종료(스냅샷 저장 후)하고 새 세션을 시작한다.

### 9.4 스냅샷 생성 실패

스냅샷 저장에 실패해도 세션 종료는 계속 진행한다. 단, 복원 시 이전 스냅샷 또는 메시지 기반 즉석 생성으로 폴백한다.
