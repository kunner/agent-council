### 1.2 네트워크 경계

| 경계 | 프로토콜 | 포트 | 설명 |
|---|---|---|---|
| 브라우저 → Cloudflare | HTTPS/WSS | 443 | TLS 1.3, Cloudflare 인증서 |
| Cloudflare → Oracle Nginx | HTTPS/WSS | 443 | Cloudflare Origin Certificate |
| Nginx → Council Server | HTTP/WS | 3001 | 내부 loopback, TLS 불필요 |
| Council Server → Firebase | HTTPS | 443 | Firebase Admin SDK (service account) |
| Council Server → GitHub API | HTTPS | 443 | Personal Access Token (PAT) |
| Council Server → Anthropic API | HTTPS | 443 | BYOK 또는 서버 환경변수 |
| 브라우저 → Firebase | HTTPS | 443 | Firebase Client SDK (직접 연결) |

## 2. 컴포넌트별 책임과 의존 관계

### 2.1 Cloudflare (프론트엔드 호스팅 + CDN)

**책임:**
- React SPA 정적 파일 빌드 및 글로벌 CDN 서빙
- Git push 이벤트 감지 → 자동 빌드/배포 파이프라인
- DNS 권한 서버 역할 (council.domain.com)
- DDoS 보호 및 봇 필터링
- PR별 프리뷰 URL 자동 생성 (`https://{branch}.council.pages.dev`)

**의존 관계:**
- GitHub 저장소 (`packages/web/`) 변경 시 빌드 트리거
- 빌드된 파일: Firebase SDK, Zustand, React Router, shadcn/ui 등이 번들에 포함
- 런타임에 브라우저가 Firebase Firestore/Auth에 직접 연결 (Council Server 경유 없음)

**배포 단위:** `packages/web/` 디렉토리 단독 빌드 (`pnpm build --filter web`)

### 2.2 Oracle Cloud Ampere — Nginx

**책임:**
- TLS 종료 (Let's Encrypt 인증서, certbot 자동 갱신 90일)
- 요청 라우팅: `/api/*` → Council Server, `/ws` → WebSocket 업그레이드
- 요청 로깅 (access.log → Grafana Loki)
- 정적 파일은 Nginx에서 서빙하지 않음 (Cloudflare Pages 담당)

**주요 Nginx 설정:**
```nginx
server {
    listen 443 ssl;
    server_name council.domain.com;

    ssl_certificate     /etc/letsencrypt/live/council.domain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/council.domain.com/privkey.pem;

    # Council Server API
    location /api/ {
        proxy_pass http://127.0.0.1:3001;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # WebSocket (타이핑 인디케이터, 실시간 세션 상태)
    location /ws {
        proxy_pass http://127.0.0.1:3001;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 86400;  # 24시간 (WebSocket 유지)
    }
}
```

**의존 관계:**
- Oracle 서버 인스턴스에 직접 설치 (Docker 외부)
- Council Server 프로세스가 :3001에서 응답해야 동작

### 2.3 Council Server (핵심 오케스트레이터)

**기술:** Node.js 20 LTS + TypeScript 5.x, Fastify 4.x, Docker (restart policy로 프로세스 생명주기 관리)

**책임 분류:**

```
Council Server 책임 영역
│
├── 요청 처리 계층 (packages/server/src/)
│   ├── HTTP API (Fastify)
│   │   ├── POST /api/projects          프로젝트 생성
│   │   ├── GET  /api/projects/:id      프로젝트 상세
│   │   ├── POST /api/projects/:id/sets Set 추가
│   │   ├── POST /api/messages          PM 메시지 전송
│   │   ├── POST /api/tasks             태스크 생성
│   │   ├── PATCH /api/tasks/:id        태스크 상태 변경
│   │   └── POST /api/git/merge         PR 머지 실행
│   │
│   └── WebSocket 서버 (ws 라이브러리)
```

**의존 관계:**
- Firebase Admin SDK → Google Cloud IAM (service account JSON)
- `simple-git` npm 패키지 → 서버 파일시스템의 git 바이너리
- `ws` npm 패키지 → HTTP 서버 upgrade 이벤트
- `claude` CLI 바이너리 → Anthropic API (BYOK)
- GitHub API → 저장소 쓰기 권한 PAT

### 2.4 Firebase Firestore (영속 데이터 + 실시간 동기화)

**책임:**
- Council Room 메시지 영구 저장 (내구성 보장)
- 클라이언트 실시간 동기화: `onSnapshot` 리스너로 메시지 추가 즉시 모든 탭에 푸시
- 프로젝트/Set/태스크/PR 메타데이터 저장
- 오프라인 지원: 네트워크 복구 시 자동 재동기화

**Firestore가 처리하는 것 vs 처리하지 않는 것:**

| 처리 O | 처리 X |
|---|---|
| 메시지 저장/조회 | Claude 세션 상태 (휘발성) |
| 태스크 CRUD | 타이핑 인디케이터 |
| Set 상태 메타데이터 | WebSocket 연결 상태 |
| 스냅샷 저장 | Git 파일 내용 |
| 사용자 프로필/API 키 | 빌드 로그 스트리밍 |

**의존 관계:**
- Council Server: Firebase Admin SDK로 쓰기
- Web UI: Firebase Client SDK로 읽기 + 실시간 리스너
- Firestore Security Rules로 사용자 격리 (각 user는 자신의 projects만 접근 가능)

### 2.5 Firebase Auth

**책임:**
- Google OAuth 2.0 / GitHub OAuth 소셜 로그인
- JWT 토큰 발급 (idToken)
- 토큰 검증 위임 (Council Server가 Admin SDK로 검증)

**인증 흐름:**
```
브라우저                Firebase Auth           Council Server
    │                        │                       │
    │── signInWithPopup() ──►│                       │
    │◄── idToken (JWT) ──────│                       │
    │                        │                       │
    │── POST /api/... ───────────────────────────────►│
    │   Authorization: Bearer {idToken}              │
    │                                                 │── verifyIdToken() ──►
    │                                                 │◄── uid, email ───────
    │                                                 │
    │◄── 200 OK ──────────────────────────────────────│
```

**의존 관계:**
- Web UI: `firebase/auth` 패키지
- Council Server: `firebase-admin/auth` verifyIdToken()
- Firestore Security Rules: `request.auth.uid` 기반 접근 제어

### 2.6 Firebase Storage

**책임:**
- Claude Code가 생성한 코드 파일 아티팩트 임시 저장
- Set 내부 작업 로그 파일 (용량이 큰 경우 Firestore 대신)
- 첨부파일 업로드/다운로드 (PM이 파일 첨부 시)

**의존 관계:**
- Council Server: Admin SDK로 업로드 (Claude 응답에서 추출한 파일)
- Web UI: 다운로드 URL 직접 접근

### 2.7 Web UI (React SPA)

**기술:** React 18 + TypeScript, Vite 5, shadcn/ui + Tailwind CSS

**책임:**
- Firebase Auth 로그인 흐름 처리
- Firestore `onSnapshot`으로 실시간 메시지 수신 및 렌더링
- Council Server WebSocket으로 타이핑 인디케이터, 세션 상태 수신
- PM 메시지 입력 → Council Server HTTP API 호출
- 칸반 보드 드래그앤드롭 (로컬 상태 변경 → API 동기화)
- Git 상태 패널 (Council Server API로 조회)

**의존 관계:**
- Firebase Client SDK (직접, Council Server 경유 없음)
- Council Server REST API (메시지 전송, 태스크 수정, Git 작업)
- Council Server WebSocket (실시간 휘발성 상태)

### 3.2 상세: Claude CLI Adapter 응답 파싱 (Phase 1)

```typescript
// packages/server/src/adapters/ClaudeCliAdapter.ts

interface ClaudeCliResponse {
  type: 'result' | 'error'
  result?: string
  usage?: { input_tokens: number; output_tokens: number }
}

async sendMessage(setId: string, message: string): Promise<string> {
  const set = await this.setService.getSet(setId)
  const worktreePath = set.worktreePath  // /workspace/{projectId}/set-a

  const args = [
    '--message', message,
    '--output-format', 'json',
    '--cwd', worktreePath,
    '--no-interactive',
  ]

  // ANTHROPIC_API_KEY를 환경에 주입 (BYOK 복호화 후)
  const env = {
    ...process.env,
    ANTHROPIC_API_KEY: await this.decryptApiKey(set.ownerId),
  }

  return new Promise((resolve, reject) => {
    const proc = spawn('claude', args, { env, cwd: worktreePath })
    let stdout = ''
    let stderr = ''

    proc.stdout.on('data', (chunk) => stdout += chunk)
    proc.stderr.on('data', (chunk) => stderr += chunk)
    proc.on('close', (code) => {
      if (code !== 0) {
        reject(new Error(`claude exited ${code}: ${stderr}`))
        return
      }
      const parsed: ClaudeCliResponse = JSON.parse(stdout)
      resolve(parsed.result ?? '')
    })
  })
}
```

### 3.3 Firestore 실시간 리스너 — 클라이언트 수신 흐름

```typescript
// packages/web/src/hooks/useMessages.ts

export function useMessages(projectId: string, roomId: string) {
  const [messages, setMessages] = useState<Message[]>([])

  useEffect(() => {
    const q = query(
      // Firestore 경로: projects/{projectId}/rooms/{roomId}/messages
      collection(db, `projects/${projectId}/rooms/${roomId}/messages`),
      orderBy('timestamp', 'asc')
    )

    // onSnapshot: Firestore가 변경사항을 브라우저에 실시간 푸시
    // 새 메시지 추가 시 ~100ms 내에 수신됨
    const unsubscribe = onSnapshot(q, (snapshot) => {
      snapshot.docChanges().forEach((change) => {
        if (change.type === 'added') {
          const msg = { id: change.doc.id, ...change.doc.data() } as Message
          setMessages((prev) => [...prev, msg])
        }
        if (change.type === 'modified') {
          setMessages((prev) =>
            prev.map((m) => (m.id === change.doc.id
              ? { id: change.doc.id, ...change.doc.data() as Message }
              : m))
          )
        }
      })
    })

    return () => unsubscribe()  // 컴포넌트 언마운트 시 리스너 해제
  }, [projectId, roomId])

  return messages
}
```

### 4.2 각 통신 방식의 사용 기준

#### REST HTTP (Fastify)

- **사용 경우:** 상태 변경을 유발하는 모든 PM 액션
- **특징:** 요청-응답 패턴, 멱등성 가능, 에러 핸들링 명확
- **구체적 사용처:**
  - `POST /api/messages` — PM 메시지 전송
  - `POST /api/projects` — 프로젝트 생성
  - `POST /api/projects/:id/sets` — Set 추가
  - `PATCH /api/tasks/:id` — 태스크 상태 변경
  - `POST /api/git/merge` — PR 머지 실행
  - `GET /api/projects/:id/status` — 프로젝트 현재 상태 조회

#### WebSocket (ws 라이브러리)

- **사용 경우:** 저장하지 않아도 되는 휘발성 실시간 이벤트
- **특징:** 양방향, 저지연, 연결 유지 오버헤드
- **구체적 사용처:**
  - 타이핑 인디케이터: `{ type: 'typing', setId, message: '생각 중...' }`
  - Set 작업 진행률: `{ type: 'progress', setId, percent: 43, task: 'API 구현' }`
  - Claude 세션 heartbeat: `{ type: 'heartbeat', setId, alive: true }`
  - Git 충돌 즉시 알림: `{ type: 'conflict', setId, files: ['src/App.tsx'] }`

**WebSocket 이벤트 타입 정의:**
```typescript
// packages/shared/src/types/ws.ts

type WsEvent =
  | { type: 'typing';    setId: string; message: string }
  | { type: 'progress';  setId: string; percent: number; task: string }
  | { type: 'heartbeat'; setId: string; alive: boolean }
  | { type: 'conflict';  setId: string; files: string[]; branch: string }
  | { type: 'session';   setId: string; status: 'restored' | 'failed' }
  | { type: 'token_usage'; setId: string; used: number; limit: number }
```

**Set 상태 ↔ Session 상태 매핑:**

<!-- Set 상태는 Firestore에 저장되는 영속 상태, Session 상태는 서버 메모리의 휘발성 상태 -->
| Set 상태 (Firestore) | Session 상태 (서버 메모리) | 설명 |
|---|---|---|
| `idle` | `STOPPED` | Set 생성 직후 또는 세션 종료 후 |
| `idle` | `CREATING` | 세션 기동 중 |
| `active` | `ACTIVE` | PM 활성, 리더 정상 응답 중 |
| `active` | `IDLE` | PM 30분 미만 부재, 세션 유지 |
| `active` | `SLEEP` | PM 30분~2시간 부재, 현재 작업만 마무리 |
| `stopping` | `STOPPING` | 스냅샷 저장 후 프로세스 종료 중 |
| `stopped` | `STOPPED` | 세션 종료 완료, PM 복귀 시 복원 가능 |

#### Firestore onSnapshot (실시간 리스너)

- **사용 경우:** 영속 데이터의 실시간 동기화
- **특징:** Firebase가 변경사항을 클라이언트에 자동 푸시, 오프라인 지원 내장
- **구체적 사용처:**
  - Council Room 메시지 스트림
  - Set 상태 변경 (idle → working → done)
  - 태스크 보드 상태 변경
  - PR 상태 변경 (open → reviewing → merged)

**리스너 구독 패턴:**
```typescript
// 메시지: 실시간 구독 (projects/{projectId}/rooms/{roomId}/messages)
onSnapshot(
  query(collection(db, `projects/${projectId}/rooms/${roomId}/messages`), orderBy('timestamp')),
  callback
)

// Set 상태: 단일 문서 감시
onSnapshot(doc(db, `projects/${projectId}/sets/${setId}`), callback)

// 태스크 보드: 필터링된 컬렉션 감시
onSnapshot(
  query(
    collection(db, `projects/${projectId}/tasks`),
    where('status', 'in', ['in_progress', 'review'])
  ),
  callback
)
```

#### subprocess (Council Server 내부)

- **사용 경우:** Claude Code CLI 호출, git 명령 실행
- **특징:** 단방향 (spawn), stdout 버퍼링, 비동기 완료 이벤트
- **구체적 사용처:**
  - `spawn('claude', ['--message', '...', '--output-format', 'json'])`
  - `spawn('git', ['worktree', 'add', path, branch])`
  - `spawn('git', ['rebase', 'origin/main'])`
  - `spawn('tmux', ['new-session', '-d', '-s', setId])`

#### MCP Channel (Phase 2 — 목표 아키텍처)

- **사용 경우:** 장기 실행 Claude Code 세션과의 양방향 통신
- **특징:** 세션 컨텍스트 유지, 스트리밍 응답, Agent Teams 내부 통신
- **구현 방향:**
  - 커스텀 MCP Channel 서버를 Council Server 내부 모듈로 구현
  - 각 Set에 대해 영구 채널 수립
  - Claude Code가 도구를 호출하면 Council Server가 처리 후 결과 반환
  - Phase 1 CLI 어댑터와 동일한 인터페이스를 유지하는 Adapter 패턴 사용

## 5. 기술 스택 선정 근거

### 5.1 서버 런타임: Node.js 20 LTS

**선정 근거:**
- Claude Code SDK/CLI가 Node.js 기반 → 동일 런타임으로 subprocess 오버헤드 최소화
- TypeScript 풀스택 (Web UI와 타입 공유 가능, `packages/shared/`)
- WebSocket 라이브러리 생태계 성숙도 (ws, socket.io 등)
- Firebase Admin SDK가 Node.js 공식 지원
- ARM (Oracle Ampere) 빌드 안정성이 Python/Java보다 우수

**대안 검토:**
- Python/FastAPI: Claude SDK Python 지원도 좋으나, TypeScript 타입 공유 불가
- Go: 성능은 우수하나 Firebase Admin SDK 미공식, 팀 학습 비용
- Bun: 빠르지만 ARM 안정성 미검증

### 5.2 HTTP 프레임워크: Fastify 4.x

**선정 근거:**
- Express 대비 2~3배 처리량 (JSON 직렬화 최적화)
- TypeScript 제네릭 스키마 검증 내장 (zod 연동)
- 플러그인 아키텍처로 인증 미들웨어, 로깅 분리 용이
- `@fastify/websocket` 플러그인으로 HTTP/WS 동일 서버에서 처리

**대안 검토:**
- Express: 레거시, TypeScript 타입 지원 후처리 방식
- Hono: 엣지 런타임 최적화이나 Oracle ARM 서버에는 이점 없음

### 5.3 Firebase: Auth + Firestore + Storage

**선정 근거 (비용):**
- Spark 플랜(무료)으로 프로젝트 전체 운영 가능
- 월 비용 $0 (초기 개발 단계에서 운영비 0원이 핵심 제약 조건)

**선정 근거 (기술):**
- Firestore `onSnapshot`: WebSocket 서버 구현 없이 실시간 메시지 동기화
  - 브라우저가 Firestore에 직접 연결 → Council Server 부하 감소
  - 오프라인 캐시 내장 → 네트워크 불안정 환경에서도 동작
- Firebase Auth: OAuth 2.0 구현 없이 Google/GitHub 소셜 로그인
- Security Rules: 서버 없이 데이터 접근 제어 (ownerId 기반 격리)

**대안 검토:**
- Supabase: PostgreSQL 기반, Realtime도 있으나 무료 플랜이 Firebase보다 제한적
- PlanetScale/Neon: DB만 제공, Auth/Storage 별도 구현 필요
- 자체 PostgreSQL: Oracle 서버에 설치 가능하나 백업/복구 운영 부담

### 5.4 프론트엔드: React 18 + Vite 5

**선정 근거:**
- Claude Code 자체가 React/TypeScript 코드 생성을 가장 잘 지원 (코드 생성 품질)

- Next.js: SSR 필요 없음 (SPA로 충분), Cloudflare Pages 설정 복잡
- SvelteKit: 번들 크기 우수하나 Claude Code 생성 품질 떨어짐

**선정 근거:**

```
→ 현재 규모(단일 사용자 또는 소규모 팀)에서는 문제 없음
```

**병목 3 — Firestore 무료 한도**
```
일 50,000 읽기 / 20,000 쓰기:
- 메시지 1개 수신 = 1 읽기 (onSnapshot 트리거)
- 활발한 Council 세션(시간당 100 메시지) = 일 2,400 읽기
→ 20개 동시 세션도 48,000 읽기 < 50,000 한도
→ 한도 근접 시 Blaze 종량제 전환 ($0.06/10만 읽기)
```

### 6.2 수평 확장 시나리오 (Phase 3 이후)

수요가 증가하여 다중 사용자/대규모 프로젝트를 지원해야 할 경우:

#### 옵션 A — Oracle Ampere 스케일업

```
현재:  1× A1.Flex (4 OCPU / 24GB)
확장:  1× A1.Flex (16 OCPU / 96GB) — ARM 최대 구성
비용:  여전히 Always Free (4 OCPU / 24GB 기준, 추가분은 유료)
```

#### 옵션 B — Council Server 수평 분산

```
                    ┌─────────────────────────────┐
                    │  Load Balancer              │
                    │  (Cloudflare Tunnel 또는    │
                    │   Oracle LB)               │
                    └──────┬──────────────────────┘
                           │
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
  │ Council     │  │ Council     │  │ Council     │
  │ Server #1   │  │ Server #2   │  │ Server #3   │
  │ (Node.js)   │  │ (Node.js)   │  │ (Node.js)   │
  └─────────────┘  └─────────────┘  └─────────────┘
```

**수평 확장 시 해결해야 할 문제:**

1. **WebSocket 세션 고정 (Sticky Session)**
   - 문제: WebSocket 연결이 특정 서버에 묶여 있음
   - 해결: Cloudflare Load Balancer의 IP Hash 기반 라우팅 또는
           Redis Pub/Sub으로 WebSocket 이벤트 브로드캐스트

2. **Claude 세션과 서버 인스턴스 매핑**
   - 문제: Set A의 Claude 프로세스가 서버 #1에 있으면 서버 #2는 접근 불가
   - 해결: 서버당 담당 프로젝트/Set을 고정 분배 (프로젝트를 서버에 핀)
           또는 분산 세션 브로커 도입 (Redis + SSH 원격 제어)

3. **파일시스템 워크스페이스 공유**
   - 문제: /workspace/{projectId}/는 특정 서버의 로컬 디스크
   - 해결: NFS 마운트 (Oracle Storage 연동) 또는
           프로젝트별 서버 고정 할당

#### 옵션 C — 클라우드 네이티브 이관

세션 수가 수십 개를 초과하는 경우:
```
- Council Server → GCP Cloud Run (컨테이너, 자동 스케일링)
- Claude 세션 → GCP Compute Engine 개별 VM (세션 격리 강화)
- 파일시스템 → GCS (Google Cloud Storage) + Cloud Filestore
- Firestore는 그대로 (이미 GCP 기반)
- 예상 비용: 세션당 ~$0.5~1/시간
```

### 6.3 세션 수 증가 대응 전략

**단기 (1~5개 프로젝트, 1~4명 사용자):**
- 현재 아키텍처 그대로 Oracle 단일 인스턴스
- PM 부재 2시간 후 세션 자동 종료로 메모리 회수
- tmux에서 세션 수 모니터링 (`tmux ls | wc -l`)

**중기 (5~20개 프로젝트, 5~20명 사용자):**
- Oracle Ampere 스케일업 (4→16 OCPU, 24→96GB)
- Firestore 쓰기 배치 처리 (메시지 10개씩 묶어서 쓰기)
- 클라이언트 캐싱 강화 (Firestore 오프라인 캐시 우선 조회)

**장기 (20개+ 프로젝트, 다중 사용자):**
- 옵션 C (GCP 클라우드 네이티브) 이관
- 사용자별 격리된 서버 인스턴스 (멀티 테넌트 SaaS 구조)
- Firestore → Blaze 종량제 자동 전환

### 6.4 단일 장애점 (SPOF) 분석

| 컴포넌트 | SPOF 여부 | 대응 |
|---|---|---|
| Oracle Ampere 인스턴스 | SPOF | Docker restart policy 자동 재시작; 장기적으로 Oracle HA 구성 |
| Firestore | SPOF 아님 | GCP 멀티존 자동 복제 |
| Cloudflare Pages | SPOF 아님 | Cloudflare 글로벌 PoP |
| Firebase Auth | SPOF 아님 | GCP 글로벌 가용성 |
| Claude API (Anthropic) | SPOF | 요청 재시도 (지수 백오프), 세션 상태 Firestore에 보존 |
| Nginx | SPOF (서버 다운 시) | systemd로 재시작; 서버 자체 장애는 Oracle 책임 |

## 8. 보안 설계

### 8.1 API 키 보안 (BYOK)

```
사용자가 API 키 등록
    │
    ├── [1] Web UI: POST /api/settings/api-key
    │         body: { apiKey: "sk-ant-..." }
    │         Authorization: Bearer {Firebase idToken}
    │
    ├── [2] Council Server:
    │       - Firebase idToken 검증 → uid 추출
    │       - AES-256-GCM 암호화:
    │           key = process.env.ENCRYPTION_KEY (32바이트 랜덤, 서버 환경변수)
    │           iv = crypto.randomBytes(12)
    │           encrypted = aes256gcm.encrypt(apiKey, key, iv)
    │       - Firestore: users/{uid}.apiKeyEncrypted = base64(iv + encrypted + authTag)
    │
    └── [3] Claude 세션 시작 시:
            - Firestore에서 암호화된 키 로드
            - 메모리에서만 복호화 (환경변수 ENCRYPTION_KEY 사용)
            - subprocess 환경변수에 주입: ANTHROPIC_API_KEY={plaintext}
            - 복호화된 평문은 메모리에서 즉시 해제 (GC에 맡김)
            - 절대 로그에 남기지 않음 (Fastify redact 설정)
```

### 8.2 Firestore Security Rules 설계 원칙

```javascript
// 핵심 원칙:
// 1. 모든 문서는 ownerId 필드를 가짐
// 2. 요청자의 uid와 ownerId가 일치해야만 쓰기 가능
// 3. 읽기는 인증된 사용자만 (로그인 필수)
// 4. 시스템 메시지 write는 Council Server만 (Admin SDK = Security Rules 우회)

// Admin SDK는 Security Rules를 우회함
// → Council Server가 Firestore에 쓰는 모든 데이터는 서버 자체적으로 검증 필요
// → idToken 검증으로 요청자 확인 후 ownerId 기반 접근 제어를 서버 코드에서도 수행
```

### 8.3 Git 자격증명 보안

```
GitHub PAT (Personal Access Token):
- Firestore: projects/{id}/git/config.githubToken = AES-256 암호화 저장
- GitHub API 호출 시에만 메모리에서 복호화
- 최소 권한 원칙: repo 스코프만 (public_repo 또는 private repo 접근)
- 서버 환경변수 ENCRYPTION_KEY와 동일한 키로 암호화
```

## 10. 구현 단계별 아키텍처 변경 사항

### Phase 1 (MVP) — 현재 계획

- Claude 연동: CLI subprocess (`ClaudeCliAdapter`)
- 세션 유지: tmux 기본 (복원 전략 B만 구현)
- WebSocket: 타이핑 인디케이터만

- 자동 Set 구성 (목표 분석 → 팀 자동 생성)
- CI/CD 연동 (빌드/테스트 자동 실행)
