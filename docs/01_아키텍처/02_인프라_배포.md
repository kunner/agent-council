## 2. Oracle Cloud Ampere 서버 구성

### 2.1 인스턴스 스펙 (Always Free Tier)

Oracle Cloud Always Free 프로그램은 Ampere A1 인스턴스를 영구 무료로 제공한다.
최대 4 OCPU + 24GB RAM까지 무료이며, 단일 인스턴스로 모두 할당하는 것을 권장한다.

| 항목 | 스펙 |
|---|---|
| 아키텍처 | ARM (AArch64 / Ampere A1) |
| OCPU | 4 |
| RAM | 24GB |
| 부팅 볼륨 (SSD) | 200GB |
| 네트워크 | 최대 4 Gbps |
| 공인 IP | 1개 (고정) |
| 리전 | ap-tokyo-1 권장 (한국 사용자 지연 최소화) |

### 2.2 ARM 아키텍처 주의사항

Oracle Ampere는 x86이 아닌 ARM이다. 모든 Docker 이미지와 바이너리는 반드시 `linux/arm64` 용으로 빌드해야 한다.

```bash
# 이미지 빌드 시 플랫폼 명시
docker buildx build --platform linux/arm64 -t council-server:latest .

# 기반 이미지도 arm64 지원 여부 확인
# node:20-alpine → arm64 지원 O
# 일부 커뮤니티 이미지 → arm64 미지원 가능, 반드시 확인
```

### 2.3 디스크 파티션 계획 (200GB SSD)

```
/               40GB   OS + 시스템
/var/lib/docker 60GB   Docker 이미지, 볼륨, 레이어
/opt/agent-council 90GB   Git 워크트리 (프로젝트별 클론)
/opt/backups    10GB   설정 파일, 스냅샷 백업
```

파티션 설정 (인스턴스 생성 후 cloud-init 또는 수동):

```bash
# /workspace 전용 파티션 마운트 예시 (블록 볼륨 추가 시)
sudo mkfs.ext4 /dev/sdb
sudo mkdir -p /workspace
echo '/dev/sdb /workspace ext4 defaults,nofail 0 2' | sudo tee -a /etc/fstab
sudo mount -a
```

### 2.4 초기 서버 셋업 스크립트

```bash
#!/usr/bin/env bash
# setup-server.sh - Oracle Ampere 초기 설정 (Ubuntu 22.04 ARM)
set -euo pipefail

# 1. 시스템 업데이트
sudo apt-get update && sudo apt-get upgrade -y

# 2. 필수 패키지
sudo apt-get install -y \
  curl wget git tmux htop \
  ufw fail2ban \
  certbot python3-certbot-nginx \
  jq unzip

# 3. Docker (ARM64 지원)
curl -fsSL https://get.docker.com | sudo bash
sudo usermod -aG docker $USER
sudo systemctl enable --now docker

# 4. Docker Compose v2
sudo apt-get install -y docker-compose-plugin

# 5. Node.js 20 LTS (ARM64)
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs

# 6. pnpm
npm install -g pnpm

# 7. workspace 디렉토리
sudo mkdir -p /opt/agent-council/workspace
sudo chown $USER:$USER /opt/agent-council/workspace

# 8. UFW 방화벽 기본 설정 (상세는 섹션 10 참조)
sudo ufw allow 22/tcp
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp
sudo ufw --force enable

# 9. fail2ban (SSH 브루트포스 방어)
sudo systemctl enable --now fail2ban

echo "초기 설정 완료"
```

### 2.5 RAM 할당 계획 (24GB)

```
OS + 커널                  ~1.0GB
Nginx                      ~0.2GB
Council Server (Node.js)   ~1.5GB   (기본 heap + 연결 풀)
Uptime Kuma                ~0.3GB
Claude Code 세션 x4        ~6.0GB   (세션당 ~1.5~2GB, 동시 4 Set 기준)
OS 버퍼/캐시               ~4.0GB
여유 (스파이크 대응)        ~11.0GB
─────────────────────────────────
합계                       ~24GB
```

동시 Set 수를 늘릴 경우 세션당 RAM 사용량을 모니터링하여 최대 Set 수 결정.

## 3. Firebase 프로젝트 설정

### 3.1 프로젝트 생성 및 서비스 활성화

Firebase Console(https://console.firebase.google.com)에서 신규 프로젝트 생성:

```
프로젝트명: agent-council
프로젝트 ID: agent-council-prod  (전역 고유값)
Google Analytics: 선택사항 (비활성화 권장 - 무료 한도 절약)
```

활성화할 서비스:

| 서비스 | 활성화 방법 | 용도 |
|---|---|---|
| Authentication | Build → Authentication → Get started | 사용자 로그인 |
| Firestore Database | Build → Firestore Database → Create database | 채팅/메타데이터 |
| Storage | Build → Storage → Get started | 첨부 파일 |

### 3.2 Firebase Auth 설정

```
Authentication → Sign-in method 탭:

활성화할 제공업체:
  ✅ Google    (Google 계정 소셜 로그인)
  ✅ GitHub    (GitHub OAuth 앱 연동 필요)

GitHub OAuth 앱 설정:
  1. GitHub → Settings → Developer settings → OAuth Apps → New OAuth App
  2. Homepage URL: https://council.yourdomain.com
  3. Authorization callback URL:
     https://agent-council-prod.firebaseapp.com/__/auth/handler
  4. Client ID/Secret을 Firebase Auth GitHub 제공업체에 입력
```

**Spark 플랜 무료 한도 (Auth):**
- 월 10,000 MAU (소셜 로그인: Google/GitHub)
- 소규모 팀 사용 시 충분

### 3.3 Firestore Database 설정

```
Firestore Database → Create database:
  모드: Production mode (보안 규칙 적용)
  리전: asia-northeast3 (서울)  ← 한국 서비스 최적화, 지연 최소화
```

**복합 인덱스 설정** (`firebase/firestore.indexes.json`):

```json
{
  "indexes": [
    {
      "collectionGroup": "messages",
      "queryScope": "COLLECTION",
      "fields": [
        { "fieldPath": "timestamp", "order": "ASCENDING" }
      ]
    },
    {
      "collectionGroup": "tasks",
      "queryScope": "COLLECTION",
      "fields": [
        { "fieldPath": "status", "order": "ASCENDING" },
        { "fieldPath": "createdAt", "order": "DESCENDING" }
      ]
    },
    {
      "collectionGroup": "logs",
      "queryScope": "COLLECTION",
      "fields": [
        { "fieldPath": "timestamp", "order": "DESCENDING" }
      ]
    }
  ],
  "fieldOverrides": []
}
```

**Spark 플랜 무료 한도 (Firestore):**

| 항목 | 무료 한도 | 예상 일 사용량 | 여유도 |
|---|---|---|---|
| 문서 읽기 | 50,000/일 | ~3,000~8,000 | 충분 |
| 문서 쓰기 | 20,000/일 | ~500~2,000 | 충분 |
| 문서 삭제 | 20,000/일 | ~100 | 충분 |
| 저장 용량 | 1GB | ~50MB (초기) | 충분 |
| 네트워크 | 10GB/월 | ~500MB | 충분 |

> 주의: 메시지 실시간 리스너(onSnapshot)는 문서 변경 시마다 읽기 1회로 집계됨.
> 클라이언트 탭 여러 개 열면 읽기가 배수로 증가하므로 탭 관리 주의.

### 3.4 Firebase Storage 설정

```
Storage → Get started:
  보안 규칙: production mode
  리전: asia-northeast3
```

스토리지 보안 규칙 (`storage.rules`):

```javascript
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {
    // 인증된 사용자만 자기 프로젝트 디렉토리에 읽기/쓰기
    match /projects/{userId}/{projectId}/{allPaths=**} {
      allow read, write: if request.auth != null
        && request.auth.uid == userId
        && request.resource.size < 50 * 1024 * 1024; // 50MB 제한
    }
    // 공개 읽기 없음 (모두 인증 필요)
  }
}
```

**Spark 플랜 무료 한도 (Storage):**
- 저장 용량: 5GB
- 다운로드: 1GB/일
- 업로드: 무제한 (저장 한도 내)

### 3.5 Firebase Admin SDK 서비스 계정

서버에서 Firestore에 쓰기/읽기하려면 Admin SDK 자격증명이 필요하다.

```
Firebase Console → 프로젝트 설정 → 서비스 계정
→ "새 비공개 키 생성" → JSON 다운로드
→ 파일명: firebase-service-account.json
→ 서버의 /opt/council/secrets/ 에 저장 (0600 권한)
→ Docker 컨테이너에 볼륨 마운트 또는 환경변수로 주입
```

`.env` 예시:
```bash
GOOGLE_APPLICATION_CREDENTIALS=/run/secrets/firebase-sa.json
# 또는 JSON 내용을 직접 환경변수로 (인코딩)
FIREBASE_SERVICE_ACCOUNT_BASE64=<base64 encoded JSON>
```

## 4. Cloudflare 설정

### 4.1 DNS 설정

도메인을 Cloudflare에 등록하거나 네임서버를 Cloudflare로 변경:

```
도메인 등록기에서 네임서버를 다음으로 변경:
  alice.ns.cloudflare.com
  bob.ns.cloudflare.com
```

Cloudflare DNS 레코드:

| 타입 | 이름 | 값 | 프록시 | 용도 |
|---|---|---|---|---|
| A | `api.council` | `<Oracle 서버 공인 IP>` | 프록시 ON (주황 구름) | Council Server API/WS |
| CNAME | `council` | `<Pages 프로젝트>.pages.dev` | 프록시 ON | Cloudflare Pages (Web UI) |
| A | `@` | `<Oracle 서버 공인 IP>` | 프록시 ON | 루트 도메인 |
| CNAME | `www` | `yourdomain.com` | 프록시 ON | www 리다이렉트 |

> `council.yourdomain.com` → Cloudflare Pages (React SPA 정적 서빙)
> `api.council.yourdomain.com` → Oracle Ampere 서버 (Council Server API + WebSocket)

### 4.2 SSL/TLS 설정

```
Cloudflare → SSL/TLS → Overview:
  암호화 모드: Full (strict)
  ← "Flexible"은 Cloudflare ↔ Origin 구간이 HTTP로 노출되므로 반드시 사용 금지
  ← "Full (strict)"는 Origin 서버에 유효한 SSL 인증서 필요 (Let's Encrypt로 해결)

SSL/TLS → Edge Certificates:
  ✅ Always Use HTTPS
  ✅ Automatic HTTPS Rewrites
  최소 TLS 버전: TLS 1.2
  ✅ TLS 1.3
  ✅ HSTS (max-age=31536000, includeSubDomains)
```

### 4.3 Cloudflare Pages 설정 (Web UI 배포)

```
Cloudflare → Pages → Create a project → Connect to Git
  리포지토리: github.com/yourorg/agent-council
  프로덕션 브랜치: main
  빌드 설정:
    빌드 명령어: pnpm --filter web build
    빌드 출력 디렉토리: packages/web/dist
    루트 디렉토리: /  (모노레포 루트)

환경 변수 (Pages 대시보드에서 설정):
  VITE_FIREBASE_API_KEY=...
  VITE_FIREBASE_AUTH_DOMAIN=agent-council-prod.firebaseapp.com
  VITE_FIREBASE_PROJECT_ID=agent-council-prod
  VITE_FIREBASE_STORAGE_BUCKET=agent-council-prod.appspot.com
  VITE_FIREBASE_MESSAGING_SENDER_ID=...
  VITE_FIREBASE_APP_ID=...
  VITE_API_BASE_URL=https://api.council.yourdomain.com
  VITE_WS_URL=wss://api.council.yourdomain.com/ws
```

커스텀 도메인 연결:
```
Pages → 프로젝트 → Custom domains → Add custom domain
  도메인: council.yourdomain.com  ← Web UI (SPA) 전용
  → Cloudflare가 CNAME 레코드를 자동으로 관리
  (API 서버는 api.council.yourdomain.com 사용 — Pages 아님)
```

**Cloudflare Pages 무료 한도:**
- 빌드: 월 500회
- 배포: 무제한
- 요청: 무제한
- 대역폭: 무제한

### 4.4 Cloudflare 캐싱 규칙

정적 자산은 캐싱하고, API/WebSocket은 캐싱하지 않도록 설정:

```
Cloudflare → Rules → Cache Rules:

규칙 1: API 및 WebSocket 캐싱 제외
  조건: URI path starts with /api OR URI path starts with /ws
  동작: Bypass cache

규칙 2: 정적 자산 장기 캐싱
  조건: URI path matches *.js, *.css, *.woff2, *.png
  동작: Cache everything, Edge TTL = 1년
```

### 4.5 WAF / DDoS 보호

Cloudflare 무료 플랜에서 기본 제공:
- DDoS 자동 완화 (Layer 3/4/7)
- 악성 봇 기본 차단
- Rate Limiting: 무료 플랜은 Rules를 통해 기본 설정 가능

```
Cloudflare → Security → Bots:
  Bot Fight Mode: ON (무료)

Cloudflare → Security → WAF:
  Managed Rules: 무료 플랜 기본 규칙 활성화
```

## 5. Nginx 리버스 프록시

### 5.1 설치

```bash
sudo apt-get install -y nginx
sudo systemctl enable --now nginx
```

### 5.2 메인 설정 파일

`/etc/nginx/nginx.conf`:

```nginx
user www-data;
worker_processes auto;  # ARM CPU 코어 수에 맞게 자동 설정
pid /run/nginx.pid;
include /etc/nginx/modules-enabled/*.conf;

events {
    worker_connections 1024;
    use epoll;
    multi_accept on;
}

http {
    # 기본 설정
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout 65;
    types_hash_max_size 2048;
    server_tokens off;  # Nginx 버전 노출 방지

    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    # 로깅
    log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                    '$status $body_bytes_sent "$http_referer" '
                    '"$http_user_agent" "$http_x_forwarded_for"';
    access_log /var/log/nginx/access.log main;
    error_log /var/log/nginx/error.log warn;

    # Gzip 압축
    gzip on;
    gzip_vary on;
    gzip_proxied any;
    gzip_comp_level 6;
    gzip_types text/plain text/css application/json application/javascript
               text/xml application/xml application/xml+rss text/javascript;

    # 클라이언트 제한
    client_max_body_size 50M;  # 파일 업로드 최대 크기

    include /etc/nginx/conf.d/*.conf;
    include /etc/nginx/sites-enabled/*;
}
```

### 5.3 Council Server 가상 호스트

`/etc/nginx/sites-available/council`:

```nginx
# HTTP → HTTPS 리디렉션
server {
    listen 80;
    listen [::]:80;
    # API 서버 도메인 (Oracle 서버가 직접 처리)
    server_name api.council.yourdomain.com;

    # Let's Encrypt ACME 챌린지는 HTTP로 허용
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    # 그 외 모두 HTTPS로 리디렉션
    location / {
        return 301 https://$host$request_uri;
    }
}

# HTTPS 메인 서버
server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    # API 서버 도메인 (Oracle 서버가 직접 처리)
    # Web UI (council.yourdomain.com)는 Cloudflare Pages가 서빙
    server_name api.council.yourdomain.com;

    # Let's Encrypt SSL 인증서

    add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # (Cloudflare IP 범위에서만 CF-Connecting-IP 신뢰)
    set_real_ip_from 103.21.244.0/22;
    set_real_ip_from 103.22.200.0/22;

    real_ip_header CF-Connecting-IP;

    location /api/ {
        proxy_pass http://127.0.0.1:3001;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        proxy_buffering off;
    }

    location /api/health {

        access_log off;

    # Uptime Kuma 대시보드 (선택사항, 내부 접근용)
    location /status/ {

        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        # IP 제한 (본인 IP만 접근 허용)
        allow <your_home_ip>;
        deny all;
    }
```

## 8. GitHub Actions CI/CD 파이프라인

### 8.1 파이프라인 전체 흐름

```
git push origin main
    │
    ├─→ Job 1: test
    │   - pnpm install
    │   - TypeScript 타입 검사
    │   - 단위 테스트
    │
    ├─→ Job 2: build-web (test 통과 시)
    │   - Cloudflare Pages가 자동 빌드/배포 처리
    │   (GitHub Actions에서 별도 처리 불필요)
    │
    └─→ Job 3: deploy-server (test 통과 시, main 브랜치만)
        - Docker 이미지 빌드 (ARM64)
        - GHCR에 이미지 푸시
        - Oracle 서버 SSH 접속
        - docker compose pull && up -d
        - 헬스체크 확인
```

### 8.2 GitHub Actions 워크플로우

`.github/workflows/deploy.yml`:

```yaml
name: Deploy

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}/council-server

jobs:
  # ──────────────────────────────────────────────
  # Job 1: 테스트 및 타입 검사
  # ──────────────────────────────────────────────
  test:
    name: Test & Type Check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v3
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Type check
        run: pnpm --filter server tsc --noEmit

      - name: Run tests
        run: pnpm --filter server test --run

  # ──────────────────────────────────────────────
  # Job 2: Docker 이미지 빌드 & 푸시 (ARM64)
  # ──────────────────────────────────────────────
  build-and-push:
    name: Build Docker Image (ARM64)
    runs-on: ubuntu-latest
    needs: test
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      # ARM64 빌드를 위한 QEMU 에뮬레이터
      - name: Set up QEMU
        uses: docker/setup-qemu-action@v3
        with:
          platforms: arm64

      # Buildx (멀티 아키텍처 빌드 지원)
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=sha,prefix=sha-
            type=raw,value=latest

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          file: packages/server/Dockerfile
          platforms: linux/arm64
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          # 레이어 캐싱 (빌드 시간 단축)
          cache-from: type=gha
          cache-to: type=gha,mode=max

  # ──────────────────────────────────────────────
  # Job 3: Oracle Ampere 서버 배포
  # ──────────────────────────────────────────────
  deploy:
    name: Deploy to Oracle Ampere
    runs-on: ubuntu-latest
    needs: build-and-push
    environment: production  # GitHub Environments 보호 규칙 적용

    steps:
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.ORACLE_SERVER_HOST }}
          username: ${{ secrets.ORACLE_SERVER_USER }}
          key: ${{ secrets.ORACLE_SSH_PRIVATE_KEY }}
          script: |
            set -e

            cd /opt/council

            # 최신 이미지 태그로 환경변수 업데이트
            IMAGE_TAG="sha-$(echo ${{ github.sha }} | cut -c1-7)"
            sed -i "s/^IMAGE_TAG=.*/IMAGE_TAG=${IMAGE_TAG}/" .env

            # GHCR 로그인 (GitHub Actions 토큰은 만료됨 → PAT 사용)
            echo "${{ secrets.GHCR_PAT }}" | docker login ghcr.io -u ${{ github.actor }} --password-stdin

            # 이미지 풀 및 재시작
            docker compose pull council-server
            docker compose up -d council-server

            # 헬스체크 대기 (최대 60초)
            echo "헬스체크 대기..."
            for i in $(seq 1 12); do
              if curl -sf http://localhost:3001/api/health > /dev/null; then
                echo "서버 정상 기동 확인"
                exit 0
              fi
              echo "대기 중... (${i}/12)"
              sleep 5
            done

            echo "헬스체크 실패 - 배포 실패"
            docker compose logs council-server --tail=50
            exit 1

      - name: Notify deployment
        if: always()
        run: |
          STATUS="${{ job.status }}"
          echo "배포 결과: ${STATUS}"
          # 필요 시 Slack/Discord 알림 훅 추가
```

### 10.1 포트 구성

| 포트 | 서비스 | 노출 범위 | 설명 |
|---|---|---|---|
| 22 | SSH | 공개 (IP 제한 권장) | 서버 관리 |
| 80 | Nginx | 공개 | HTTP → HTTPS 리디렉션 |
| 443 | Nginx | 공개 | HTTPS 메인 (API + WebSocket) |
| 3001 | Council Server | 로컬호스트 only | Nginx가 프록시 |
| 3002 | Uptime Kuma | 로컬호스트 only | Nginx /status/ 경로로 접근 |
| 9100 | Node Exporter | 로컬호스트 only | Prometheus 스크랩 |

### 10.2 UFW 방화벽 규칙

```bash
# 기본 정책: 인바운드 거부, 아웃바운드 허용
sudo ufw default deny incoming
sudo ufw default allow outgoing

# SSH (본인 IP로만 제한 권장)
sudo ufw allow from <your_ip> to any port 22
# 또는 전체 허용 (보안 위험 있음)
# sudo ufw allow 22/tcp

# HTTP/HTTPS (Cloudflare IP만 허용하면 더 안전)
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp

# UFW 활성화
sudo ufw enable
sudo ufw status verbose
```

**강화 옵션 - Cloudflare IP만 허용:**
```bash
# Cloudflare IP 목록으로만 80/443 허용 (DDoS 직접 공격 차단)
for ip in \
  103.21.244.0/22 103.22.200.0/22 103.31.4.0/22 \
  104.16.0.0/13 104.24.0.0/14 108.162.192.0/18 \
  131.0.72.0/22 141.101.64.0/18 162.158.0.0/15 \
  172.64.0.0/13 173.245.48.0/20 188.114.96.0/20 \
  190.93.240.0/20 197.234.240.0/22 198.41.128.0/17; do
    sudo ufw allow from $ip to any port 80
    sudo ufw allow from $ip to any port 443
done
```

### 10.4 fail2ban 설정

SSH 브루트포스 자동 차단:

`/etc/fail2ban/jail.local`:
```ini
[DEFAULT]
bantime  = 3600   ; 1시간 차단
findtime = 600    ; 10분 내
maxretry = 5      ; 5회 실패 시 차단

[sshd]
enabled = true
port    = 22
logpath = /var/log/auth.log
```

```bash
sudo systemctl enable --now fail2ban
sudo fail2ban-client status sshd  # 차단 현황 확인
```
